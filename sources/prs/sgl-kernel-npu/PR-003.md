# sgl-kernel-npu — Fused GDN Gating + `solve_tril` Performance

## Overview

PR #466 ("improve performance for fused gdn gating and solve tril" by zhaozx-cn) tunes two pieces of sgl-kernel-npu's FLA (Flash Linear Attention) path for the Gated DeltaNet (GDN) model:

1. **`fused_gdn_gating`** — a Triton-on-Ascend kernel that fuses the GDN gating elementwise chain (softplus → exp → mul, plus a sigmoid output) into one launch, with Unified-Buffer-aware block sizing.
2. **`solve_tril`** — blocked triangular-matrix-inverse kernels (`solve_tril_16x16` → `merge_16x16_to_32x32`/`_64x64`) used by the delta-rule recurrence.

This page documents the PR as an upstream-code source. The interesting Ascend-specific bits are: deriving tile sizes from the on-chip Unified Buffer size, switching the grid shape based on whether the batch fits in the core count, and offering a `_without_sigmoid` variant so the sigmoid can be fused into a *downstream* kernel instead.

## The GDN Gating Computation

GDN (gated delta net) gating produces two outputs per `(batch, head)`:

- `g = -exp(A_log) * softplus_beta(a + dt_bias)`  — the gate
- `beta_output = sigmoid(b)`  — the output gate

where `softplus_beta(x) = (1/β)·log(1 + exp(β·x))` for `β·x ≤ threshold`, else `x` (the threshold branch avoids `exp` overflow — the standard softplus stability trick). Done naively this is four or five separate kernels (add bias, softplus, exp, mul, sigmoid), each reading and writing the `[batch, num_heads]` tensor through Global Memory. The fused kernel does all of it in one pass:

```python
x = blk_a.to(tl.float32) + blk_bias.to(tl.float32)[None, :]
softplus_x = tl.where(
    beta * x <= threshold, (1 / beta) * tl.log(1 + tl.exp(beta * x)), x
)
blk_g = -tl.exp(blk_A_log.to(tl.float32)) * softplus_x
tl.store(g + off, blk_g.to(g.dtype.element_ty), mask=mask)

blk_beta_output = tl.sigmoid(blk_b.to(tl.float32))
tl.store(beta_output + off, blk_beta_output.to(beta_output.dtype.element_ty), mask=mask)
```

Everything stays in registers/UB between operations — `softplus_x`, `blk_g`, `blk_beta_output` are all intermediate local tiles, never spilled to GM. This is `technique-operator-fusion` applied to an elementwise chain: the win is N-1 fewer GM round-trips for an N-op chain.

## UB-Aware Block Sizing

The host wrapper `fused_gdn_gating_npu` does the Ascend-specific tuning. It queries the device core count and sizes the batch tile `BLK_BATCHES` from the Unified Buffer budget:

```python
num_cores = get_device_properties()[1]
BLK_HEADS = 8
elem_size = a.element_size()
max_ub_batches = int((UNIFIED_BUFFER_SIZE * 0.95) / (BLK_HEADS * elem_size))
if batch <= num_cores:
    progs = batch
    BLK_BATCHES = 1
    ROW_ITER = 1
else:
    progs = num_cores
    FACTOR = 8 * num_heads
    calc_blk_batches = (
        triton.next_power_of_2(
            triton.cdiv(int(UNIFIED_BUFFER_SIZE * 0.95), FACTOR * BLK_HEADS * elem_size)
        ) // 2
    )
    BLK_BATCHES = max(1, min(calc_blk_batches, max_ub_batches, 64))
    row_per_core = triton.cdiv(batch, progs)
    ROW_ITER = triton.cdiv(row_per_core, BLK_BATCHES)
```

Several Ascend tuning ideas are packed in here:

- **UB budget as the hard ceiling.** `max_ub_batches = UB * 0.95 / (BLK_HEADS * elem_size)` is the largest batch tile that fits in the Unified Buffer (with a 5% safety margin). The actual `BLK_BATCHES` is then `min(calc_blk_batches, max_ub_batches, 64)` — never exceed UB, also cap at 64 for occupancy. Sizing tiles to the on-chip buffer is the Ascend analogue of sizing CUDA shared-memory tiles; oversizing spills to GM and undoes the fusion.
- **Two regimes by batch size.** When `batch <= num_cores`, every batch row gets its own program (`progs = batch`, `BLK_BATCHES = 1`) — no tiling needed, full parallelism. When batch exceeds the core count, the grid is pinned to `num_cores` programs and each program loops over `ROW_ITER` batch tiles of `BLK_BATCHES` rows. This avoids launching `batch` programs when `batch ≫ cores` (which would just serialize on the cores anyway) and instead gives each core a contiguous chunk of work.
- **Power-of-2 tile, halved.** `next_power_of_2(...) // 2` picks a tile that is a power of two (Triton-friendly) but deliberately conservative — leave UB headroom for the multiple intermediate tiles (`softplus_x`, `blk_g`, `blk_beta_output`) the kernel materializes simultaneously.

## The `_without_sigmoid` Fusion Split

The PR also ships `fused_gdn_gating_kernel_without_sigmoid`, which computes `g` but **not** `beta_output = sigmoid(b)`. The reason is downstream fusion: if the consumer of `beta_output` is itself a kernel that can absorb a sigmoid (e.g., the delta-rule update that multiplies by the output gate), fusing the sigmoid there saves one GM write+read of the `[batch, num_heads]` tensor. This is the standard "fusion boundary" decision — push an elementwise op across the kernel boundary when the consumer can host it, keep it inside when it cannot. The two variants let the caller choose the boundary per call site.

## `solve_tril`: Blocked Triangular Inverse

The other half of the PR tunes `solve_tril`, the lower-triangular matrix inverse used by the gated delta-rule recurrence. The strategy is **blocking and merging**: invert small `16×16` diagonal blocks in parallel, then merge them up the hierarchy:

```
solve_tril_16x16_kernel_paral_v3   -> per-block 16x16 inverse
merge_16x16_to_32x32_inverse_kernel
merge_16x16_to_64x64_inverse_kernel_reorder_all_masked
```

Each merge level combines four smaller inverses (plus the off-diagonal solve) into a larger inverse, doubling the block dimension. Parallelism comes from `NT` independent `16×16` blocks across heads/batch (`solve_tril_16x16_kernel_paral_v3[NT, B * H]`), and the merge kernels consolidate them. The `_reorder_all_masked` variant reorders the merge computation so all masking is applied uniformly — a tuning that reduces divergent predication inside the merge. This blocked inverse is the linear-algebra analogue of the tiling in `technique-nz-tiling`: keep the working set in on-chip buffers (here, register/UB tiles for the small blocks) and merge upward, rather than materializing the full inverse in GM.

## Cost Model and Trade-offs

- **Fusion saves GM bandwidth, not FLOPs.** The gating chain is memory-bound (elementwise ops on a `[batch, heads]` tensor); fusion turns ~5 GM passes into 1. The speedup is bounded by the original GM traffic fraction — near-linear for a pure elementwise chain.
- **UB sizing is the lever.** Too-small `BLK_BATCHES` underutilizes UB and increases loop overhead; too-large spills. The `0.95 * UB` ceiling plus the `64` occupancy cap is the PR's chosen balance.
- **`_without_sigmoid` trades a kernel for a fusion opportunity.** It is only a win if the downstream consumer actually fuses the sigmoid; otherwise it just adds a separate sigmoid kernel. Dispatch must know the consumer.
- **`solve_tril` blocking depth.** Going to `64×64` blocks reduces the number of merge passes but raises per-block register/UB pressure. The 16→32→64 ladder is the point where the small-block inverses still fit comfortably on-core.

## Pitfalls

- **UB size is per-architecture.** `UNIFIED_BUFFER_SIZE` differs between 910 and 910B (and cores count differs). The `get_device_properties()` query is what makes this portable; hard-coding either value breaks the other platform.
- **Multiple live intermediates.** The kernel keeps `softplus_x`, `blk_g`, and `blk_beta_output` alive at once. The `0.95` margin must cover *all* of them, not one — if you extend the chain, re-derive the budget, don't assume the existing `BLK_BATCHES` still fits.
- **Threshold branch correctness.** The `beta * x <= threshold ? softplus : x` branch must match the reference exactly; the threshold is a numerical-stability switch, not an approximation. Changing it silently changes numerics.
- **Triton-on-Ascend ≠ Triton-on-CUDA.** The `tl.program_id` grid and the UB sizing assume the Triton-NPU backend maps programs to AICores. CUDA-derived Triton intuitions about occupancy do not transfer directly.
