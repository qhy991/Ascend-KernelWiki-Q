# sgl-kernel-npu — DeepEP Low-Latency CCU Offload Path

## What Changed

PR #478 adds a CCU offload path for DeepEP low-latency dispatch and combine. The PR body states that the dispatch and combine operators are implemented through the CCU path and controlled by the `MOE_ENABLE_CCU` environment variable.

The patch is large: 18 files, +4392/-67. It adds CCU-specific tiling helpers, dispatch/combine tiling implementations, AICore-side CCU headers, quantization helpers, and op API wiring.

## Why It Matters

DeepEP dispatch/combine is on the hot path of MoE serving. Moving low-latency dispatch/combine toward a CCU-backed path is a communication/runtime optimization: token routing and expert aggregation are not only memory movement problems, they are distributed coordination problems.

The PR is also valuable because it documents a limitation: the CCU interface is encapsulated by HCCL, so outer-layer calls still require data movement and the distribution result can only be produced after the HCCL-side operator completes communication. The PR body explicitly says performance had reached a bottleneck before the lower-level CCU interface was exposed, so this patch is a basic functional adaptation rather than a final performance endpoint.

## Execution Path

The modified path spans host tiling, op API, and kernel headers:

1. `csrc/deepep/deep_ep.cpp` wires DeepEP to the new mode.
2. `op_host/mc2_tiling_utils.h` and new dispatch/combine tiling helper headers provide shared tiling logic.
3. `moe_distribute_dispatch_v2_ccu_tiling.cpp` and `moe_distribute_combine_v2_ccu_tiling.cpp` build CCU-mode tiling data.
4. `op_api/aclnn_moe_distribute_dispatch_v2.cpp` and `op_api/aclnn_moe_distribute_combine_v2.cpp` route host calls.
5. `op_kernel/moe_distribute_dispatch_v2.cpp` and `op_kernel/moe_distribute_combine_v2.cpp` select CCU-specific implementations.
6. `moe_distribute_dispatch_v2_ccu.h` and `moe_distribute_combine_v2_ccu.h` contain the CCU-mode kernel logic.
7. `quantize_functions.h` supports quantized data handling used by dispatch/combine modes.

## Mechanism

The key mechanism is a mode split in DeepEP low-latency dispatch/combine:

- Normal low-latency paths keep their existing dispatch/combine implementations.
- `MOE_ENABLE_CCU` selects CCU offload behavior.
- Host tiling prepares CCU-specific metadata for both dispatch and combine.
- The op API and AICore entry points branch into CCU headers.
- Quantization helpers are factored into a shared header so CCU dispatch/combine can reuse scale/quant logic.

## Cost Model

The intended benefit is lower-latency expert dispatch/combine through a CCU-specialized path. The current cost model has an important caveat:

| Component | Expected Benefit | Current Limitation |
| --- | --- | --- |
| CCU dispatch/combine | offload communication-heavy routing work | HCCL encapsulation forces outer-layer data movement |
| Host tiling helpers | prepare mode-specific tiling once | more tiling branches and metadata to validate |
| Shared quant helpers | reuse quantization logic across modes | scale/dtype mismatch can silently corrupt routed activations |

## Trade-offs and Pitfalls

- **Not a pure performance win yet.** The PR itself states that CCU performance is bottlenecked until the lower-level interface is exposed.
- **Mode coverage is incomplete.** The PR body says cases such as `topk=-1` and `bs=0` are deferred for later CCU support.
- **Environment-gated behavior must be rank-consistent.** `MOE_ENABLE_CCU` must be applied consistently across ranks.
- **Dispatch and combine must evolve together.** A CCU dispatch path without matching combine semantics can break expert routing.
