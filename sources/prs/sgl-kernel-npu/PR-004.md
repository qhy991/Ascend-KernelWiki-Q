# sgl-kernel-npu — DeepEP Low-Latency Alltoall (`dispatch`/`combine`) on NPU

## Overview

PR #546 ("add ll alltoall" by zzx-study) refactors sgl-kernel-npu's DeepEP low-latency expert-parallel (EP) communication so that the **alltoall is encapsulated inside the Python API** rather than exposed as a standalone primitive. The PR provides `low_latency_dispatch` and `low_latency_combine` as the external entry points, backed by CANN's `torch_npu.npu_moe_distribute_dispatch_v2` / `npu_moe_distribute_combine_v2` ops, and adds an `alltoall` strategy name to the low-latency strategy map. It fixes issue #554.

This page documents the PR as an upstream-code source. The Ascend-specific substance is how MoE expert-parallel alltoall maps onto **CANN's fused `npu_moe_distribute_*` ops** (which fold the scatter/gather, token routing, and HCCL alltoall into one kernel), and how a Python strategy layer selects between communication algorithms (`fullmesh_v1`/`fullmesh_v2`/`hierarchy`) for low-latency decoding.

## Why "Low-Latency" MoE Alltoall

In expert-parallel MoE, each rank holds a subset of experts. A forward step must:

1. **Dispatch**: send each token to the rank(s) owning its chosen experts (an alltoall, plus the routing/index bookkeeping), receive the gathered expert inputs.
2. **Compute** the expert FFN on the gathered tokens.
3. **Combine**: send expert outputs back to the tokens' origin ranks (the reverse alltoall), weighted-sum the top-k outputs.

For *batched* (prefill/throughput) MoE this is a large, bandwidth-saturating alltoall — DeepEP's "normal" mode. For *low-latency decode* the token count per rank is tiny (often single-digit), so the alltoall is latency-bound, not bandwidth-bound: the per-message setup cost and the number of synchronization points dominate. Low-latency mode optimizes for that regime by fusing the routing + communication into a single CANN op and minimizing host-side orchestration. This is the same normal-vs-low-latency split DeepEP makes on NVIDIA; the PR ports the low-latency path to Ascend via CANN ops.

## The Strategy Layer

The PR adds `alltoall` to the low-latency strategy map (`ep_strategy.py`) alongside the normal-mode strategies:

```python
class LowLatencyStrategy:
    ALLTOALL = "alltoall"
    ...

class StrategyMap:
    strategy_map = {
        ...
        ("alltoall"): (NormalStrategy.ALLTOALL, LowLatencyStrategy.ALLTOALL),
    }
```

`StrategyMap.get_strategy(deep_mode=...)` resolves the mode string to a strategy name, and the buffer (`deep_ep/buffer.py`) instantiates the matching `LowLatencyEPCommStrategy` subclass. Two concrete strategies ship:

- `DefaultLowLatencyCommStrategy` — delegates to `self.runtime.low_latency_dispatch/combine` (the runtime's native path).
- `OpsLowLatencyCommStrategy` — the NPU path, calling `_npu_low_latency_dispatch` / `_npu_low_latency_combine`, which wrap the CANN ops. Its `comm_alg` defaults to `"hierarchy"`.

The strategy abstraction (PR #540 introduced the strategy split; #546 adds the alltoall variant) is what lets the same Python `low_latency_dispatch`/`combine` API target different backends without the caller caring which CANN op or algorithm runs underneath.

## The CANN Ops Behind the API

The NPU strategy's dispatch is a single call into CANN:

```python
(
    expand_x, dynamic_scales, expand_idx, expert_token_nums,
    ep_recv_counts, tp_recv_counts, expand_scales,
) = torch_npu.npu_moe_distribute_dispatch_v2(
    x=x,
    expert_ids=topk_idx,
    group_ep=self.group_name,
    ep_world_size=self.group_size,
    ep_rank_id=self.rank,
    moe_expert_num=num_experts,
    scales=x_smooth_scale,
    x_active_mask=x_active_mask,
    expert_scales=topk_weights,
    expert_shard_type=expert_shard_type,
    shared_expert_num=shared_expert_num,
    shared_expert_rank_num=shared_expert_rank_num,
    quant_mode=quant_mode,
    expert_token_nums_type=expert_token_nums_type,
    global_bs=num_max_dispatch_tokens_per_rank * self.group_size,
    comm_alg=comm_alg,   # "": default, "fullmesh_v1", "fullmesh_v2", "hierarchy"
)
```

And combine is the symmetric `npu_moe_distribute_combine_v2`. Several things are notable:

- **The alltoall is *inside* the op.** `npu_moe_distribute_dispatch_v2` is a fused CANN kernel: it does the per-token expert routing, the data scatter, the HCCL alltoall across the EP group, and the receive-side gather — all in one launch. The Python layer never calls a bare `alltoall`; it calls the MoE-distribute op, which internally performs it. This is the "encapsulated alltoall" the PR title refers to.
- **Routing metadata comes back.** The op returns `expert_token_nums`, `ep_recv_counts`/`tp_recv_counts`, `expand_idx`, and `expand_scales` — the bookkeeping the combine step needs to reverse the dispatch. Carrying these through the op (rather than recomputing them in Python) avoids a redundant scan over the routing indices.
- **Quantization is in the comm path.** `quant_mode` / `dynamic_scales` / `expand_scales` mean the dispatch can quantize tokens for transmission and the combine can dequantize — fusing the quant into the alltoall to cut inter-rank bandwidth. This matters in low-latency decode where the alltoall is the bottleneck.
- **`global_bs = num_max_dispatch_tokens_per_rank * group_size`** fixes the dispatch buffer shape to a worst-case token budget per rank, so the op uses static-shaped buffers (no per-step shape negotiation — a latency win, at the cost of padding).

## `comm_alg`: Choosing the Communication Topology

The `comm_alg` parameter selects how the alltoall is realized over the HCCS fabric:

- `""` — default CANN algorithm.
- `fullmesh_v1` / `fullmesh_v2` — full-mesh point-to-point exchanges (each rank talks directly to every other rank). Good when the EP group spans NPUs connected by HCCS with enough links to sustain the mesh; the `_v1`/`_v2` variants differ in scheduling/packing.
- `hierarchy` — hierarchical exchange (e.g., intra-node first, then inter-node). Better for multi-node EP where a flat fullmesh would saturate inter-node links.

The PR enforces a constraint: `comm_alg="hierarchy"` requires `shared_expert_num == 0`. Shared experts (always-executed experts not routed to) have their own combine path that the hierarchical algorithm does not support, so the assert prevents a silent misconfiguration.

## Cost Model and Trade-offs

- **Latency vs. bandwidth.** Low-latency mode trades bandwidth efficiency (static `global_bs` padding, per-rank worst-case buffers) for fewer synchronization points and a single fused launch. For batch-1/low-token decode this is the right trade; for prefill it wastes bandwidth — use the normal mode there.
- **Fusion removes host overhead.** Fusing routing + alltoall + gather into one CANN op removes the Python dispatch and intermediate GM tensors a decomposed implementation would create. In low-latency decode, host-side overhead is a large fraction of the step, so this is where the win is.
- **Quant-in-comm.** Fusing quantization into the dispatch/combine halves inter-rank traffic when enabled, but adds dequant compute on the receive side. The `quant_mode` switch lets the caller decide per deployment.
- **Algorithm choice is topology-dependent.** `fullmesh` and `hierarchy` are not interchangeable; the right one depends on whether the EP group is intra-node (HCCS full mesh) or cross-node (hierarchy to spare inter-node bandwidth). Picking wrong silently hurts latency.

## Pitfalls

- **`hierarchy` + shared experts is rejected.** The assert is explicit, but if you bypass the strategy layer and call the op directly with `comm_alg="hierarchy"` and shared experts, behavior is undefined — the op may not enforce it.
- **`global_bs` sizing.** `num_max_dispatch_tokens_per_rank` must be an upper bound on per-rank dispatched tokens; undersizing truncates tokens silently, oversizing wastes memory and bandwidth on padding. It is a static contract, recomputed per step in Python but fixed for the op call.
- **Environment-variable coupling.** `MOE_SHARED_EXPERT_RANK_NUM` and `MOE_EXPERT_TOKEN_NUMS_TYPE` are read from env inside the strategy method. Setting them inconsistently across ranks breaks the alltoall bookkeeping — they are collective invariants, not per-rank knobs.
- **`group_ep` must match the EP process group.** The CANN op uses `group_ep` (the HCCL group name) for the alltoall. A mismatched or uninitialized group produces a silent wrong-rank exchange. This is the same hazard as any HCCL collective — see `technique-hccl-optimization`.
- **Low-latency ≠ normal mode.** The dispatch/combine return shapes and metadata differ from the normal DeepEP path. Don't swap strategies expecting identical outputs; the buffer layer's `low_latency_*` and normal APIs are deliberately separate.
