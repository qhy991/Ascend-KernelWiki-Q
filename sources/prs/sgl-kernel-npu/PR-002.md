# sgl-kernel-npu ‚Äî LoRA Kernels on the CUBE Unit (`sgemmc`)

## Overview

PR #432 ("LoRA: Implementing kernels using CUBE computation unit" by vlserov) adds a new family of LoRA kernels ‚Äî `sgemmc_shrink` and `sgemmc_expand` ‚Äî to sgl-kernel-npu. The "c" in `sgemmc` is the point: these kernels run the low-rank LoRA matmul on the **Cube unit** through `AscendC::Matmul`, instead of computing it as elementwise multiply + reduction on the **Vector unit** like the existing `bgmv_*` / `sgmv_*` variants.

This page documents the PR as an upstream-code source. It is a clean illustration of a recurring Ascend decision: for a given low-rank GEMM, do you spend the **Vector** unit (SIMD, good at elementwise/reduce, no MAC array) or the **Cube** unit (MAC array, built for matrix multiply)? The PR's contribution is making the Cube option available and selecting it where it wins.

## LoRA Math and Why the Unit Choice Matters

A LoRA adapter for a linear layer `W` (frozen) adds `ŒîW = B @ A`, where `A` shrinks `[hidden] ‚Üí [r]` and `B` expands `[r] ‚Üí [hidden]`, with rank `r ‚â™ hidden`. Per token, the adapter computes two matmuls:

1. **Shrink**: `x_shrunk = x @ A·µÄ`, shape `[hidden] ‚Üí [r]` (one token, so really a vector-matrix product).
2. **Expand**: `y = x_shrunk @ B·µÄ`, shape `[r] ‚Üí [hidden]`, scaled by `lora_scale`.

There are two ways to realize a vector-matrix product on Ascend:

- **Vector path (`bgmv`/`sgmv`):** cast to fp32, do elementwise `Mul(x_tile, w_tile)`, then `ReduceSum` to collapse the inner dimension, accumulating into the output. No MAC array is used; the inner-product is reconstructed from SIMD ops.
- **Cube path (`sgemmc`):** treat the token's `[1, K]` activation and the `[r, K]` weight as genuine matrices and call `AscendC::Matmul`, which streams them through the Cube MAC array exactly like a full GEMM.

The Vector path is fine when `r` is tiny and the per-call overhead of setting up Cube (tiling, L0 staging) dominates. The Cube path wins as soon as the inner work is large enough to amortize that setup, because the MAC array delivers matrix-multiply throughput the Vector unit physically cannot match. The PR makes both available so the caller can pick by shape.

## The Cube Kernel: `sgemmc_shrink`

The Cube variant declares its operands through `AscendC::MatmulType` and an `AscendC::Matmul` object:

```cpp
using X_MAT_TYPE = AscendC::MatmulType<AscendC::TPosition::GM,    CubeFormat::VECTOR, X_T, false>;
using W_MAT_TYPE = AscendC::MatmulType<AscendC::TPosition::GM,    CubeFormat::ND,     W_T, true>;  // transposed
using Y_MAT_TYPE = AscendC::MatmulType<AscendC::TPosition::VECIN, CubeFormat::ND,     INNER_T>;
using BIAS_MAT_TYPE = AscendC::MatmulType<AscendC::TPosition::GM,  CubeFormat::ND,     float>;
using MAT_TYPE = AscendC::Matmul<X_MAT_TYPE, W_MAT_TYPE, Y_MAT_TYPE, BIAS_MAT_TYPE, CFG_MDL>;
```

A few things to notice:

- **`X_MAT_TYPE` uses `CubeFormat::VECTOR`.** The activation is a single token (`[1, K]`), so it is fed as a "vector" operand ‚Äî the Cube unit's M dimension is 1. The weight is `CubeFormat::ND` and transposed (`true`) so the `[r, K]` weight is consumed as `[K, r]` to match `[1, K] @ [K, r] = [1, r]`.
- **`INNER_T` accumulator.** The matmul accumulates into a wider type (`INNER_T`, typically fp32) in `Y_MAT_TYPE` positioned at `VECIN`, then the epilogue casts back to the output dtype.
- **`CFG_MDL`** is the matmul config (tiling/MdL template parameter) selected at compile time.

The per-core work and the actual multiply:

```cpp
REGIST_MATMUL_OBJ(pipe_, GetSysWorkSpacePtr(), matmulObj, &tiling);
matmulObj.DisableBias();
matmulObj.SetWorkspace(workspaceGlobal);
matmulObj.SetOrgShape(tiling.M, tiling.N, tiling.Ka, tiling.Kb);
matmulObj.SetSingleShape(tiling.singleCoreM, reqLoRARank_, tiling.singleCoreK);
matmulObj.SetTensorA(xInGm_[tokenIdx * inputHiddenDim_], false);
matmulObj.SetTensorB(wInGm_[reqLoRAWeightOffset_ + sliceIdx * inputHiddenDim_ * reqLoRARank_], true);
matmulObj.template Iterate<false>();
```

`Iterate<false>` runs the Cube matmul without iterating the result-fetch internally (the `false` means "don't auto-repeat"); the kernel then pulls tiles out with `GetTensorC` in its own loop so it can apply the LoRA scale and cast before writing back:

```cpp
AscendC::Muls(tmpTensor, mmResTensor, reqLoRAScale_, elements);   // scale by lora_scale
AscendC::PipeBarrier<PIPE_V>();
AscendC::Cast(output, tmpTensor, AscendC::RoundMode::CAST_RINT, elements);  // fp32 -> output dtype
```

So the kernel is a **Cube matmul + Vector epilogue (scale + cast)** pipeline ‚Äî `Matmul` for the heavy arithmetic, Vector for the cheap elementwise finish. This is `technique-cube-vector-overlap` in miniature: the two units do different phases of the same op.

### Multi-token batching via `BlockIterator`

`sgemmc` is invoked across all AICores (and AIV cores, with an early-return for sub-block 1). Each core derives `(tokenIdx, sliceIdx)` from its `blockIdx`, then resolves which request/LoRA adapter that token belongs to via a small helper:

```cpp
lora_common::BlockIterator blockIterator(seqLenGm_);
int64_t requestBlock = blockIterator.GetBlockIdx(tokenIdx);
reqLoRAIndex_ = loraIndicesGm_.GetValue(requestBlock);
reqLoRAWeightOffset_ = reqLoRAIndex_ * singleLoRAWeightLen_;
reqLoRARank_        = loraRanksGm_.GetValue(reqLoRAIndex_);
reqLoRAScale_       = loraScalesGm_.GetValue(reqLoRAIndex_);
```

`BlockIterator` (in `lora_common_kernel.h`) walks a cumulative-sequence-length array to map a flat token index to its request, caching the previous block for sequential access. This is the standard "variable-length request ‚Üí flat token grid" mapping used by paged/batched serving. Once the request is known, the kernel reads that request's LoRA rank, scale, and weight offset, and skips cores whose rank is 0 (`if (reqLoRARank_ == 0) return;`) ‚Äî idle adapters cost nothing.

## The Vector Kernel: `bgmv_shrink` for Contrast

The pre-existing `bgmv_shrink` computes the same `x @ A·µ¢·µÄ` without the Cube unit. Its inner tile is:

```cpp
Cast(xTmpTensor, xLocal, ...);   // -> fp32
Cast(wTmpTensor, wLocal, ...);   // -> fp32
pipe_barrier(PIPE_V);
Mul(wTmpTensor, xTmpTensor, wTmpTensor, numElements);   // elementwise product
pipe_barrier(PIPE_V);
ReduceSum<float>(wTmpTensor, wTmpTensor, wTmpTensor, numElements);  // collapse inner dim
pipe_barrier(PIPE_V);
acc += wTmpTensor.GetValue(0);
```

This is a hand-rolled dot product: cast to fp32, multiply elementwise, reduce-sum, accumulate. It uses only the Vector unit's SIMD lanes; the Cube MAC array sits idle. For very small `r` and small `K` tiles this can be competitive (no Cube tiling/setup overhead, no L0 staging), but it scales poorly: the work is `O(K)` Vector ops per output element with no hardware matrix-multiply acceleration.

## When Cube Wins

The selection between `sgemmc` (Cube) and `bgmv`/`sgmv` (Vector) is fundamentally a break-even analysis:

- **Cube fixed cost.** `AscendC::Matmul` pays for tiling setup, L0A/L0B staging, and the matmul-object machinery. For a `[1, K] @ [K, r]` with tiny `r` and `K`, this fixed cost can exceed the actual MAC work.
- **Cube throughput.** Once `K * r` is large enough, the MAC array delivers far more multiply-accumulate work per cycle than Vector `Mul`+`ReduceSum`.
- **Rank and batch.** `sgmv` (batched, multiple tokens sharing an adapter) and `sgemmc` both amortize better than `bgmv` (single token). The Cube path shines for larger ranks and batched groups.

The PR does not hard-code the switch; it provides the Cube kernels (`sgemmc_shrink`/`sgemmc_expand`) so the Python layer can route by shape. (Note: #432 was landed, briefly reverted in #430, and re-introduced ‚Äî the revert/reland cycle is worth checking before depending on a specific commit.)

## Trade-offs and Pitfalls

- **Cube setup overhead for tiny ranks.** Routing a `r=4` adapter through `sgemmc` may be slower than `bgmv`. The dispatch must be shape-aware, not "always Cube."
- **M=1 operand format.** The activation is `CubeFormat::VECTOR` (M=1). Cube matmuls with M=1 underutilize the MAC array's M dimension; the win comes from K and N being large enough, not from M. Don't expect full Cube utilization on single-token LoRA.
- **Epilogue on Vector.** The scale + cast live on the Vector unit after `GetTensorC`. If you fuse more (e.g., the expand's activation), keep the Cube‚ÜíVector handoff explicit with `PipeBarrier<PIPE_V>` as the kernel does; dropping the barrier races the two units.
- **Workspace.** `matmulObj.SetWorkspace(workspaceGlobal)` allocates a per-core workspace tile (`blockIdx * maxElements`). The host must size this ‚Äî undersizing corrupts adjacent cores' tiles.
- **`DisableBias()`.** LoRA shrink/expand has no bias; the bias matmul type is declared for API completeness but disabled. Don't accidentally enable it.
- **Revert history.** Confirm the kernel is present on your branch (the #430 revert); the technique is valid regardless, but the symbol may not be exported on every commit.
