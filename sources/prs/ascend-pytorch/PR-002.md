# torch_npu — npu_quant_matmul Dynamic W8A8 Operator

## Overview

This PR adds native W8A8 (8-bit weight, 8-bit activation) support to `torch_npu` through two custom operators: `npu_dynamic_quant`, which produces per-token INT8 activations with a matching scale tensor, and `npu_quant_matmul` (the quantized matmul, often abbreviated QMM), which runs an INT8 x INT8 General Matrix Multiply on the Cube unit and dequantizes the result. Together they are the canonical building block that downstream W8A8 inference stacks — Ascend-vLLM and SGLang — call for quantized linear layers on Ascend 910B.

## Operator Surface

### `npu_quant_matmul`

The quantized matmul takes two INT8 operands plus the scaling metadata needed to bring the INT32 accumulator back into a floating-point output dtype.

```python
import torch
import torch_npu

# Weights are pre-quantized to INT8 and kept resident on-device in NZ layout.
# x1: activations, INT8, shape [M, K]
# x2: weights,     INT8, shape [K, N] (stored NZ on the NPU)
out = torch_npu.npu_quant_matmul(
    x1,                       # INT8 activations  [M, K]
    x2,                       # INT8 weights      [K, N]
    scale,                    # per-channel weight scale (dequant)
    offset=None,             # optional zero-point offset
    pertoken_scale=None,     # per-token activation scale (from dynamic quant)
    bias=None,               # optional FP/INT bias added after dequant
    output_dtype=torch.float16,
)
```

The full positional signature consumed by the inference stacks is:

```python
npu_quant_matmul(x1, x2, scale, offset=None, pertoken_scale=None,
                 bias=None, output_dtype=torch.float16)
```

### `npu_dynamic_quant`

Activations are not quantized offline — they are quantized on the fly per token, so each row of the activation matrix gets its own scale. This keeps the dynamic range tight for outlier-heavy transformer activations without a calibration pass.

```python
# x: high-precision activations [M, K] (e.g. fp16 / bf16)
x_int8, scale = torch_npu.npu_dynamic_quant(x)
# x_int8: INT8 [M, K]
# scale : per-token (per-row) float scale, length M

out = torch_npu.npu_quant_matmul(
    x_int8, weight_int8, weight_scale,
    pertoken_scale=scale,
    output_dtype=torch.float16,
)
```

## Compute Path

The operator implements the standard symmetric W8A8 GEMM contract:

1. **Quantize activations** — `npu_dynamic_quant` reduces each token row to its absolute max, derives a per-token scale, and casts to INT8 on the Vector unit (`hw-vector-unit`).
2. **INT8 x INT8 matmul** — the Cube unit (`hw-cube-unit`) multiplies INT8 operands and accumulates in **INT32**, which avoids overflow across the K reduction without the Cube ever touching floating point.
3. **Dequantize** — the INT32 accumulator is scaled back by the outer product of the per-token activation scale (`pertoken_scale`) and the per-channel weight `scale`, with optional `offset` and `bias`, and emitted in `output_dtype` (fp16 here).

Symbolically, for accumulator `acc_int32 = sum_k(x1_int8 · x2_int8)`:

```
out[m, n] = acc_int32[m, n] * pertoken_scale[m] * scale[n] + bias[n]
```

## Weight Layout

Weights stay **INT8 in NZ format** resident on the device for the lifetime of the model. The Cube unit consumes the fractal NZ tiling natively, so there is no per-call NZ↔ND conversion on the weight side — only the activation path crosses the quantize boundary each step. See `technique-format-conversion` and `technique-nz-tiling` for the layout mechanics, and `pattern-nz-format-traps` for the layout mistakes this avoids.

## Throughput Characteristics

W8A8 doubles effective Cube MAC throughput relative to fp16 because the INT8 datapath packs two operands where fp16 packs one, and it halves the weight footprint and the weight-side HBM bandwidth. The qualitative trade-off table below summarizes the operand format choices (no absolute benchmark numbers are claimed here):

| Path                | Weight dtype | Activation dtype | Accumulator | Weight HBM | Notes                          |
|---------------------|--------------|------------------|-------------|------------|--------------------------------|
| fp16 matmul         | fp16         | fp16             | fp32        | 1.0x       | baseline (`kernel-matmul-ascendc`) |
| W8A8 QMM (this PR)  | INT8 (NZ)    | INT8 (per-token) | INT32       | 0.5x       | dynamic activation quant       |

## Trade-offs, Pitfalls, and Notes

- **Per-token, not per-tensor.** `pertoken_scale` must be the length-`M` scale returned by `npu_dynamic_quant`; reusing a single per-tensor activation scale defeats the dynamic-range benefit and degrades accuracy on outlier-heavy layers.
- **Pass the right scales to the right axis.** `scale` is the per-channel (length-`N`) weight scale; `pertoken_scale` is the per-row (length-`M`) activation scale. Swapping them silently produces wrong dequantized outputs.
- **`output_dtype` is explicit.** The operator does not infer the output precision from the inputs — the INT32 accumulator can be emitted as fp16 (shown) or other supported floating dtypes; set it deliberately.
- **`offset` is for asymmetric quant.** Leave `offset=None` for the common symmetric W8A8 path; only supply it when activations or weights use a non-zero zero-point.
- **Weights should already be INT8/NZ.** This operator does not quantize weights; that happens offline. Keeping weights INT8-NZ resident is what removes the weight-side conversion from the hot path.
- **Small-`M` (decode) regimes are memory-bound.** At batch/decode shapes where `M` is tiny the kernel is dominated by weight bandwidth, which is exactly where the halved INT8 weight footprint pays off most.
