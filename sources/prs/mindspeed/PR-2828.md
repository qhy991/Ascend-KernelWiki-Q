# MindSpeed — all_to_all_v_c Variable-Count MoE Communication Optimization

## What Changed

Gitee PR !2828 proposes all-to-all variable-count communication optimization for MindSpeed. Agent and targeted diff evidence show two changed MindSpore MoE/tensor-parallel files:

- `mindspeed/mindspore/core/tensor_parallel/mappings.py`
- `mindspeed/mindspore/core/transformer/moe/moe_feature/tp_extend_ep/token_dispatcher.py`

The Gitee page was observed on 2026-06-19 as not yet merged, so this page records proposed upstream evidence rather than merged behavior.

## Why It Matters

MoE token dispatch rarely has equal token counts per rank. A fixed-count all-to-all path can waste bandwidth or require padding. Variable-count all-to-all lets dispatch communicate the true per-rank token volume, which is especially relevant when expert load is skewed.

## Mechanism

The PR introduces a path around `torch.ops.comm_func.all_to_all_v_c`, paired with token dispatcher changes and rank/world-size caching. The important design point is that communication sizes become part of the token-dispatch contract rather than an afterthought hidden inside the collective wrapper.

## Trade-offs and Pitfalls

- **Counts must match dispatch metadata.** Token permutation, expert assignment, and all-to-all counts need the same ordering.
- **Variable-count collectives expose imbalance.** Performance may improve by removing padding, but skewed experts still create stragglers.
- **Rank/world-size caching must not hide topology changes.** Cached process-group metadata must follow the active MoE parallel group.
