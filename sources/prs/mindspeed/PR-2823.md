# MindSpeed — Bucket Group Reordering for Param-Gather Overlap

## What Changed

Gitee PR !2823 proposes a feature that records bucket-group trigger order and resets later bucket-group order to improve overlap after parameter-gather is enabled. Agent and targeted diff evidence show changes in documentation, distributed data parallel configuration, and feature registration:

- `docs/features/reset-bucket-group-order.md`
- `mindspeed/core/distributed/distributed_data_parallel.py`
- `mindspeed/core/distributed/distributed_data_parallel_config.py`
- `mindspeed/features_manager/distributed/reset_bucket_group_order_feature.py`

The Gitee page was observed on 2026-06-19 as not yet merged, so this page records proposed upstream evidence rather than merged behavior.

## Why It Matters

Distributed training overlap often depends on launch order. If parameter-gather buckets are prefetched in an order that does not match the real backward/forward trigger order, communication can land on the critical path. Recording the first iteration and replaying a better bucket order is a pragmatic scheduling technique.

## Mechanism

The feature observes bucket-group usage in an early iteration, then uses that order to reset later parameter-gather scheduling. The goal is to prefetch parameters for future computation while current computation is still running, rather than discovering the need too late.

## Trade-offs and Pitfalls

- **First-iteration order must be representative.** Dynamic graphs or changing microbatch schedules can make replayed order stale.
- **Overlap is order-sensitive.** The same communication volume can either hide under compute or stall the step depending on launch timing.
- **Feature flags need clear defaults.** Reordering distributed buckets can make debugging harder if enabled implicitly.
