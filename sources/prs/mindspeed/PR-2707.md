# MindSpeed — Pipeline Parallel P2P Uses isend/irev on NPU

## What Changed

MindSpeed PR #2707 adds an `optimize_p2p_comm` feature for pipeline parallelism on NPU. The Gitee PR page and diff show it is merged, with six changed files and a focused implementation under `mindspeed/core/pipeline_parallel/optimize_p2p_comm` and `mindspeed/features_manager/pipeline_parallel/optimize_p2p_comm.py`.

The core mechanism is to set `config.batch_p2p_comm = False`, replacing `batch_isend_irecv` style behavior with explicit `isend` / `irecv` point-to-point operations for pipeline-parallel communication on NPU.

## Why It Matters

Pipeline parallel training is sensitive to communication latency between adjacent stages. On Ascend NPU, the batched P2P path may not be optimal for every PP schedule. This PR records a backend-specific performance choice: expose a feature gate that selects non-batched P2P primitives when they perform better.

## Trade-offs

- This is a communication scheduling optimization, not a kernel implementation.
- The win is backend/topology dependent; `batch_isend_irecv` may still be appropriate elsewhere.
- Feature registration is as important as the adaptor: the optimized path must be selected consistently across PP ranks.
