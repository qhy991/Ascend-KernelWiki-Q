## Why It Matters

Grouped matmul is central to MoE expert execution. A post-matmul add is a small operation, but if it is left as a separate kernel it adds launch overhead and intermediate memory traffic. `GroupedMatmulAdd` makes this post-GMM epilogue explicit in the operator stack, similar in spirit to later GMM + SwiGLU + Quant fusion.

## Execution Path

The PR covers the full custom-op lifecycle:

1. Documentation defines operator behavior.
2. Example code shows invocation and expected inputs.
3. AICore/operator source implements the grouped matmul plus add path.
4. Host-side `ophost` files define proto, tiling, and ACLNN-style entry points.
5. UT files validate operator behavior.
6. `moe_finalize_routing_v2` changes keep downstream routing/finalization interfaces aligned.

## Mechanism

The fused op attaches an elementwise add epilogue to grouped matmul. This is smaller than a full activation/quant epilogue, but still important because grouped expert workloads can produce many segmented outputs. Keeping post-add close to grouped matmul reduces the number of intermediate tensors and standalone operator launches.

## Trade-offs and Pitfalls

- **Do not ignore routing coupling.** The PR is not only a new operator; it also adjusts MoE finalize routing files.
- **Add fusion is shape-sensitive.** Broadcast/add semantics must align with grouped expert segment layout.
- **Small epilogues still matter.** The gain is launch and memory-traffic reduction, not more Cube math.
