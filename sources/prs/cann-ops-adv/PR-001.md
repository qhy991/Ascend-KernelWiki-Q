# cann-ops-adv — `grouped_matmul_swiglu_quant` Fused MoE FFN Kernel

## Overview

PR #331 ("add gmmSwigluQuant") adds the `grouped_matmul_swiglu_quant` operator to `cann-ops-adv`. It fuses the **up-projection half of a quantized MoE FFN** into a single AscendC kernel: a grouped matmul (GateUp projection) on the Cube unit, followed by a SwiGLU activation, followed by a per-token dynamic quantization to int8 with its scale — all in one launch, with the matmul weight pre-laid-out in `FRACTAL_NZ`.

This page documents the PR as an upstream-code source. It is one of the cleanest examples in the corpus of three things at once: (1) **Cube→Vector fused epilogue** (`technique-cube-vector-overlap`), (2) **NZ-format weight feeding the Cube unit** (`technique-nz-tiling`), and (3) **end-to-end MoE FFN fusion** (`technique-operator-fusion`) where the int8 output is produced *for* the next matmul (the down-projection `w2`), so the quantization is fused into the activation epilogue rather than emitted as a separate kernel.

## Where It Fits in the MoE FFN

A quantized MoE FFN block per token group is, conceptually:

```
x              -> GroupedMatmul(x, w1)        # gate-up projection  [hidden -> 2*inter]
[gate, up]     -> SwiGLU = up * silu(gate)    # activation
swiglu_out     -> Quant(swiglu_out) -> int8   # dynamic per-token quant
int8, scale    -> GroupedMatmul(int8, w2)     # down projection      [inter -> hidden]
```

Without this op, that is three kernels with two GM round-trips of the `[tokens, 2*inter]` and `[tokens, inter]` activation tensors between them. `grouped_matmul_swiglu_quant` collapses the first three boxes — GMM(w1) + SwiGLU + Quant — into one kernel whose output is already int8 + per-token scale, ready to feed the down-projection's quantized matmul. The `[tokens, inter]` activation never lands in GM as fp16/bf16; it is produced, activated, and quantized on-core, then written out as int8 (half the bytes) plus a per-token fp32 scale.

## Operand Layouts

The matmul types declare the Cube-friendly layouts explicitly:

```cpp
template <bool trans = false>
using xType     = MatmulType<AscendC::TPosition::GM, CubeFormat::ND,  DTYPE_X>;
template <bool trans = false>
using weightType= MatmulType<AscendC::TPosition::GM, CubeFormat::NZ,  DTYPE_WEIGHT>;
using yType     = MatmulType<AscendC::TPosition::GM, CubeFormat::ND,  MM_DTYPE_Y>;
```

- **Weight is `CubeFormat::NZ`.** The expert weights are stored in `FRACTAL_NZ` so the Cube MAC array consumes them with no runtime reformat (the same load-time-layout principle as `pr-vllm-ascend-002`'s MoE `w2` defaulting to NZ). The activation `x` stays `ND`.
- **Accumulator `MM_DTYPE_Y` is fp32.** The Cube matmul accumulates the gate-up products in fp32 (`int32`-typed GM workspace is used for the raw matmul output, then promoted), which is what lets the SwiGLU and Quant stages run in fp32 without an intermediate precision-loss round-trip.

## The Three-Stage Compute Class

The kernel is a thin `extern "C"` entry that instantiates `GMMSwigluCompute` (or a split-workspace variant for larger shapes). The compute class exposes the fused pipeline as separate methods, each mapped to a hardware unit:

```cpp
class GMMSwigluCompute {
    inline void MMCompute(uint32_t groupIdx, MNConfig& mnConfig, uint32_t coreIdx);  // Cube
    inline void Swiglu(uint32_t loopIdx);      // Vector
    inline void Quant(uint32_t loopIdx);       // Vector
    inline void Dequant(uint32_t loopIdx);     // Vector (for the w2-input path / variants)
    inline void VectorCompute(uint32_t loopIdx);
    ...
};
```

`Process()` loops over the expert group list (`gmmSwiglu->groupListLen`), and for each group runs `MMCompute` then the Vector epilogue (`Swiglu` → `Quant`), threading the matmul output through `mmOutQueue` (a TQue between the Cube and Vector stages).

### Stage 1 — Grouped Matmul (Cube)

`MMCompute` drives `AscendC::Matmul` over the per-group `(M, K, N)` config derived from the group list and tiling. The weight's NZ layout means no `TransData` is inserted before the Cube unit. The fp32 result tile is enqueued to `mmOutQueue` for the Vector stage. This is `technique-multicore-grid-scheduling` over the group dimension combined with `technique-nz-tiling` for the operand.

### Stage 2 — SwiGLU (Vector)

The matmul output tile holds `[gate | up]` concatenated along the token-length axis. SwiGLU splits them and computes `up * silu(gate)`:

```cpp
LocalTensor<float> src0Local = _inMMLocal[... + tokenLen / 2];  // gate half
LocalTensor<float> src1Local = _inMMLocal[...];                  // up half
SwiGLU<float, false>(workspaceLocal, src0Local, src1Local, beta, tokenLen / 2);
PipeBarrier<PIPE_ALL>();
DataCopy(_inMMLocal[...], workspaceLocal, repeatParams);         // write swiglu back in place
```

`SwiGLU<float, false>` is the AscendC high-level API for `up * silu(gate)` (the `false` template selects the non-fused variant since the inputs are already separated). The `PipeBarrier<PIPE_ALL>` after it is the Cube→Vector handoff barrier: it ensures the Vector SwiGLU finishes before the next Cube matmul tile overwrites the shared `mmOutQueue`/workspace. This barrier is the synchronization cost of the fusion — you save GM bandwidth at the price of one in-core barrier per tile.

### Stage 3 — Per-Token Dynamic Quant (Vector)

`Quant` produces int8 + a per-token scale using the standard absmax recipe:

```cpp
Abs(half_out, full_out, tokenLen/2);                       // |x|
ReduceMaxTemplate(workspaceLocal, ...);                    // max |x| over the token
float quantScale = workspaceLocal.GetValue(0) / QUANT_SCALE_INT8;  // = max/127
quantScaleLocal.SetValue(loopIdx, quantScale);             // emit per-token scale
quantScale = 1 / quantScale;
Muls(full_out, full_out, quantScale, tokenLen/2);          // x * (127/max)  -> [-127,127]
CastFp32ToInt8Template(quantLocal, full_out, ...);         // fp32 -> int8
```

So the quant is: `Abs → ReduceMax → scale = max/127 → Muls → Cast fp32→int8`, all on the Vector unit, reusing the same `mmOutQueue` tile. The per-token scale (`max/127`) is written to `quantScaleOutput` alongside the int8 `quantOutput`, which is exactly what the down-projection's quantized matmul needs as its `pertoken_scale` (see `pr-vllm-ascend-004`'s `npu_quant_matmul` interface). The `QUANT_SCALE_INT8 = 127` and `BISECT = 2` constants reflect that the SwiGLU output is half the gate-up width (the `2*inter` → `inter` reduction).

## Why the Fusion Wins

The benefit is almost entirely GM bandwidth, not FLOPs:

| Tensor | Unfused (3 kernels) | Fused (1 kernel) |
| --- | --- | --- |
| `[tokens, 2*inter]` GMM(w1) output | written to GM, read back by SwiGLU | stays in UB (`mmOutQueue`) |
| `[tokens, inter]` SwiGLU output | written to GM, read back by Quant | stays in UB |
| `[tokens, inter]` int8 + scale | written to GM by Quant | written to GM once (final output) |

For a `2*inter = 2*11008` MoE, that is two full `[tokens, 11008]` fp16 GM round-trips eliminated per step per expert group. The compute (SwiGLU, Abs, ReduceMax, Muls, Cast) is the same either way; it just moves from standalone kernels into the Cube epilogue. The cost is the `PipeBarrier<PIPE_ALL>` synchronization between Cube and Vector stages — cheap relative to the GM traffic saved.

This is the same principle as NVIDIA's fused MoE kernels (e.g. vLLM's `fused_moe` with quantized epilogue), realized in AscendC with Cube (Matmul) + Vector (SwiGLU/Quant) + explicit `PIPE` barriers instead of CUDA's `cp.async` + `__syncthreads`.

## Trade-offs and Pitfalls

- **Cube→Vector barrier is mandatory.** The `PipeBarrier<PIPE_ALL>` between `MMCompute` and `Swiglu`/`Quant` is not optional — `mmOutQueue` is shared. Dropping it races the Cube write against the Vector read. The fusion pays this barrier per tile; if tiles are too small, barrier overhead can eat the bandwidth savings.
- **Weight must be NZ at load time.** `CubeFormat::NZ` assumes the expert weights were pre-converted. If a caller passes ND weights, the Cube unit either fails or silently inserts a slow reformat — the same trap as `pattern-nz-format-traps`. The op does not convert for you.
- **fp32 accumulator footprint.** SwiGLU and Quant run in fp32 in UB. The token tile length (`gmmSwiglu->tokenLen`) must fit the UB budget alongside the matmul's own L0/UB needs; the host tiling (`grouped_matmul_swiglu_quant_tiling.cpp`) is what enforces this. Oversized tokens spill.
- **Per-token scale is absmax/127.** The down-projection must use the matching dequant convention (`npu_quant_matmul` with `pertoken_scale`). A scale/dequant mismatch between this op's output and the w2 matmul silently corrupts numerics.
- **Split-workspace variant.** The entry point also instantiates `GMMSwigluSplitWorkSpaceCompute` for shapes that do not fit a single workspace — larger MoE configs take the split path. The two share the SwiGLU/Quant logic but differ in workspace partitioning; behavior is equivalent, performance is not.
- **int8 output is `/2` packed.** `quantOutput` is sized `M * tokenLen / 2` bytes — int8 elements packed two-per-byte-footprint is not what this says; rather it reflects the `2*inter → inter` halving after SwiGLU. Check the element count against your down-projection's expected input before wiring.
