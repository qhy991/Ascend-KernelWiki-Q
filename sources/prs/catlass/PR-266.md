## Why It Matters

CATLASS performance depends heavily on tile shapes, layouts, swizzle policies, L1/L0 residency, and operator variants. Hand-picking those combinations does not scale. This PR turns kernels into registered operations inside a manifest, then lets a tuner filter, launch, profile, and record candidates.

## Execution Path

The toolchain has two halves:

1. **Library generation.** Python search-space code builds GEMM/grouped-GEMM operation instances and emits registration code into a manifest.
2. **Runtime tuning.** The tuner loads registered operations, filters them by problem config, initializes arguments/workspace, launches candidates, clears L2 between runs when needed, profiles latency, and saves metrics.

## Mechanism

The central abstraction is `Operation`. Each generated GEMM candidate exposes description metadata and a uniform initialize/run interface. `Manifest` owns a list of operations; `OpConfig` filters them for a concrete problem; `OpLauncher` executes the selected operation with workspace; `Metrics` records the result.

This separates **what kernels exist** from **which kernel is best for this shape**.

## Trade-offs and Pitfalls

- **Search space can explode.** The diff includes generated logs showing thousands of basic matmul tile shapes and hundreds of grouped matmul tile shapes.
- **Tuning must control cache state.** The tool adds an L2-clear DFX kernel so measurements are less polluted by prior candidates.
- **Workspace is part of candidate validity.** Operation initialization and launcher paths must allocate the workspace expected by that generated operator.
- **Generated manifests need stable naming.** Metrics only help if kernel names encode enough layout/tile policy information to compare later.
