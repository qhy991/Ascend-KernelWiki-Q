# CATLASS — Basic Matmul and Group GEMM Examples

CATLASS is a CUTLASS-style C++ template library for Ascend that layers GEMM kernels as composable, statically-instantiated components rather than monolithic AscendC operators. This page walks the two anchor examples, `examples/00_basic_matmul.cpp` and `examples/16_group_gemm.cpp`, tracing the template instantiation chain from `Arch` down to the launched `Kernel`, the host AscendCL memory flow, and how `16_group_gemm` targets MoE `GroupedMatmul`.

## What CATLASS Is

Like CUTLASS on GPU, CATLASS expresses a GEMM as a stack of policy types that the compiler fuses into a single device kernel. The developer picks the architecture tag, a dispatch policy, tile shapes, and the operand `GemmType`s; the library assembles the block-level MMA loop, the data-movement (copy) stages, and the C-tile epilogue. Because every layer is a template parameter, tiling and ping-pong buffering are resolved at compile time with no virtual dispatch on the Cube path. See [migration-cutlass-to-catlass](../../wiki/migration/cutlass-to-catlass.md) for the concept-by-concept mapping from the GPU library, and [kernel-gemm-catlass-cpp](../../wiki/kernels/) for the kernel-side write-up.

## The Template Instantiation Chain

The `00_basic_matmul` example builds a kernel from the bottom up. Each layer narrows the previous one:

```cpp
#include "catlass/catlass.hpp"
#include "catlass/arch/arch.hpp"
#include "catlass/layout/layout.hpp"
#include "catlass/gemm/gemm_type.hpp"
#include "catlass/gemm/block/block_mmad.hpp"
#include "catlass/gemm/kernel/basic_matmul.hpp"
#include "catlass/gemm/dispatch_policy.hpp"

using namespace Catlass;

// 1. Architecture tag — selects the Atlas A2 (910B) Cube path
using ArchTag = Arch::AtlasA2;

// 2. Operand element + layout types
using LayoutA = layout::RowMajor;
using LayoutB = layout::ColumnMajor;
using LayoutC = layout::RowMajor;

using AType = Gemm::GemmType<half, LayoutA>;   // A: fp16, row-major
using BType = Gemm::GemmType<half, LayoutB>;   // B: fp16, column-major
using CType = Gemm::GemmType<half, LayoutC>;   // C: fp16, row-major

// 3. Tile shapes — L1 block tile and L0 (Cube) instruction tile
using L1TileShape = GemmShape<128, 256, 256>;  // M, N, K per core block
using L0TileShape = GemmShape<128, 256, 64>;   // K-split for the Cube MMA

// 4. Dispatch policy — enables L1/L0 ping-pong double buffering
using DispatchPolicy = Gemm::MmadAtlasA2Pingpong<true>;

// 5. Block-level MMA: ties policy + tiles + operand types together
using BlockMmad = Gemm::Block::BlockMmad<
    DispatchPolicy, L1TileShape, L0TileShape,
    AType, BType, CType>;

// 6. Block swizzle — maps logical (M,N) tiles to AI Cores
using BlockScheduler = Gemm::Block::GemmIdentityBlockSwizzle<>;

// 7. The launchable kernel (Mmad, Epilogue=void, Scheduler)
using MatmulKernel = Gemm::Kernel::BasicMatmul<BlockMmad, void, BlockScheduler>;
```

Reading top to bottom: `ArchTag` fixes the Cube ISA, the `GemmType`s bind element type to memory layout, the two `GemmShape` tiles set the L1 block and the L0 Cube-instruction granularity, and `MmadAtlasA2Pingpong` is what turns on the L1/L0 ping-pong (double buffer) so the next K-slice loads while the current one feeds the Cube unit. `BlockMmad` is the inner accumulation loop; `BasicMatmul` wraps it with the swizzle and the C-tile copy-out.

### Kernel Launch Syntax

CATLASS uses AscendC's triple-angle-bracket launch. The grid dimension is the number of AI Cores to occupy; `nullptr` is the (unused) L2 control argument, and `stream` is the AscendCL stream:

```cpp
// GemmCoord carries the problem shape {M, N, K}
GemmCoord problemShape{M, N, K};

MatmulKernel<<<aicCoreNum, nullptr, stream>>>(
    problemShape, gmA, gmB, gmC);
```

The kernel itself is parameter-light: the problem shape plus three global-memory pointers. All tiling, buffer sizing, and pipeline structure already live in the instantiated type.

## Host AscendCL Memory Flow

The host side is plain AscendCL (ACL). The example acquires a device and stream, allocates device buffers, copies operands up, launches, and copies the result back:

```cpp
aclInit(nullptr);
aclrtSetDevice(deviceId);
aclrtStream stream;
aclrtCreateStream(&stream);

uint8_t *gmA, *gmB, *gmC;
aclrtMalloc((void **)&gmA, sizeA, ACL_MEM_MALLOC_HUGE_FIRST);
aclrtMalloc((void **)&gmB, sizeB, ACL_MEM_MALLOC_HUGE_FIRST);
aclrtMalloc((void **)&gmC, sizeC, ACL_MEM_MALLOC_HUGE_FIRST);

aclrtMemcpy(gmA, sizeA, hostA, sizeA, ACL_MEMCPY_HOST_TO_DEVICE);
aclrtMemcpy(gmB, sizeB, hostB, sizeB, ACL_MEMCPY_HOST_TO_DEVICE);

MatmulKernel<<<aicCoreNum, nullptr, stream>>>(
    GemmCoord{M, N, K}, gmA, gmB, gmC);

aclrtSynchronizeStream(stream);
aclrtMemcpy(hostC, sizeC, gmC, sizeC, ACL_MEMCPY_DEVICE_TO_HOST);

aclrtFree(gmA); aclrtFree(gmB); aclrtFree(gmC);
aclrtDestroyStream(stream);
aclrtResetDevice(deviceId);
aclFinalize();
```

This is the same ACL skeleton used by hand-written AscendC kernels (see [kernel-matmul-ascendc](../../wiki/kernels/matmul-ascendc.md)); CATLASS only changes what runs *inside* the launch.

## Layouts, NZ Tiling, and the Cube Path

The `layout::RowMajor` / `layout::ColumnMajor` types are logical descriptors over global memory. On the Cube unit the operands are consumed in the hardware's fractal **NZ** (FRACTAL_NZ) form, and CATLASS's copy stages perform the ND→NZ staging through L1 into the L0A/L0B buffers as part of the `BlockMmad` data path. The `L0TileShape` K dimension (64 here) controls the per-MMA K-slice that the `MmadAtlasA2Pingpong` policy ping-pongs. For the tiling rationale see [nz-tiling](../../wiki/techniques/nz-tiling.md); the ping-pong buffering itself is covered in [double-buffering](../../wiki/techniques/double-buffering.md), and the unit it feeds in [cube-unit](../../wiki/hardware/cube-unit.md).

## Example 16: Group GEMM for MoE

`examples/16_group_gemm.cpp` generalizes the single-GEMM kernel to a *batch of independent GEMMs* whose per-group shapes differ — the exact shape of MoE expert FFNs, where each expert sees a different token count. Instead of one `GemmCoord`, the host passes per-group problem sizes and per-group operand pointers; the kernel iterates groups and reuses the same `BlockMmad` block-level loop for each. This is the CATLASS analogue of the `GroupedMatmul` operator and pairs with [kernel-grouped-gemm-ascendc](../../wiki/kernels/grouped-gemm-ascendc.md) and the MoE dispatch in [kernel-moe-ascendc](../../wiki/kernels/moe-ascendc.md).

```cpp
// Group GEMM packs many variable-shaped problems into one launch.
// Each group g computes  C_g = A_g x B_g  with its own GemmCoord.
using GroupMatmulKernel =
    Gemm::Kernel::GroupedMatmul<BlockMmad, void, BlockScheduler>;

GroupMatmulKernel<<<aicCoreNum, nullptr, stream>>>(
    problemCount, gmProblemShapes,   // array of per-group {M,N,K}
    gmGroupA, gmGroupB, gmGroupC);   // per-group operand pointers
```

The win for MoE is launch amortization: one kernel sweeps all experts, so the host does not pay per-expert launch and synchronization overhead, and the swizzle balances variable-sized expert tiles across AI Cores.

## How the Two Examples Compare

| Aspect | 00_basic_matmul | 16_group_gemm |
| --- | --- | --- |
| Problem shape | single `GemmCoord{M,N,K}` | array of per-group `GemmCoord` |
| Launch entry | `Kernel::BasicMatmul` | `Kernel::GroupedMatmul` |
| Operand pointers | one `gmA/gmB/gmC` triple | per-group pointer arrays |
| Reuses `BlockMmad` | yes | yes (same block loop per group) |
| Primary target | dense GEMM / linear layers | MoE expert FFNs |

Both share the `ArchTag`, `GemmType`, tile shapes, dispatch policy, and swizzle layers; only the top `Kernel` type and the host argument marshalling change.

## Trade-offs, Pitfalls, and Notes

- **Compile-time everything.** Tiles and the dispatch policy are template parameters, so changing them recompiles the kernel. This is the cost of zero runtime dispatch overhead on the Cube path; tune by re-instantiating, not by passing flags.
- **Tile shape must respect L1/L0 capacity.** Oversized `L1TileShape` overflows the L1 buffer; an overlarge `L0TileShape` K-slice defeats the `MmadAtlasA2Pingpong` ping-pong because there is no room for the second buffer. Keep the L0 K dimension small enough that two slices fit (see [double-buffering](../../wiki/techniques/double-buffering.md)).
- **Layout vs. NZ.** `layout::RowMajor`/`ColumnMajor` describe global memory only. The Cube unit still consumes FRACTAL_NZ; mismatched assumptions about where the ND→NZ conversion happens are a common source of wrong results — the copy stage in `BlockMmad` owns it.
- **`aicCoreNum` is the grid.** Passing fewer cores than the swizzle expects underutilizes the Cube array; for group GEMM, very small expert groups can leave cores idle if the swizzle cannot balance them.
- **No fabricated performance numbers.** CATLASS's value here is structural (composable, statically-fused GEMM), not a measured speedup; this page reports the API surface only, consistent with its `source-reported` grounding.
