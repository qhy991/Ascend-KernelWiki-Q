# CATLASS ‚Äî BlockMmad Preload nextBlock K-Tile Index Fix

## What Changed

Gitee PR !159 fixes a CATLASS BlockMmad preload bug. Agent-verified Gitee evidence shows it is merged, with four changed files:

- `include/catlass/gemm/block/block_mmad_gemm.hpp`
- `include/catlass/gemm/block/block_mmad_preload.hpp`
- `include/catlass/gemm/block/block_mmad_preload_tla.hpp`
- `include/catlass/gemv/block/block_gemv_aic.hpp`

## Why It Matters

Preload pipelines try to fetch the next block while the current block computes. That overlap only works if the next block's tile indices, K-loop count, and GM offsets are computed from the next block's actual shape. Reusing current-block K metadata can preload the wrong tile or read the wrong memory range.

This is a high-value correctness bugfix because it documents a common pipeline hazard: ‚Äúnext‚Äù work must use next-block metadata, not current-block loop state.

## Mechanism

The fix introduces next-block-specific values such as:

- `kLoopsNext`
- `kTileCountNext`
- `firstTileIdxNext`

It computes these from `actualShapeNext.k()` and `kNextBlockActual`, then uses the next block's actual K extent to derive GM offsets.

## Trade-offs and Pitfalls

- **Preload bugs can be silent.** Wrong next-block offset may only fail for tail K blocks or nonuniform block shapes.
- **Pipelining increases state.** Current and next block metadata must be kept separate.
- **GEMM and GEMV paths both matter.** The fix touches BlockMmad GEMM and GEMV AIC code.
