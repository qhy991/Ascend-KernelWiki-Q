## Why It Matters

Context-parallel prefill is communication-heavy: sequence partitions need attention-visible KV data, while DSA also has projection and output-mixing work that can run on local compute resources. The PR is valuable because it treats CP as a schedule, not just a collective call.

## Mechanism

The optimization centers on zigzag context parallelism for DSA prefill. The key design is to overlap communication and compute where dependencies allow it:

- reuse or align communication groups with existing TP/CP structure;
- all-gather KV data needed by later attention slices;
- overlap KV communication with local projection work such as `q_up` / `W_uk` style computation;
- merge head/tail schedule fragments where that reduces idle periods;
- keep precision tests around the changed CP attention path.

## Trade-offs and Pitfalls

- **Overlap depends on true dependencies.** Moving all-gather earlier is safe only if later compute consumes the same logical KV slices.
- **CP and TP groups can drift.** Reusing communication domains simplifies launch order but raises shape/partition assumptions.
- **Prefill is not decode.** A schedule optimized for prefill may not apply to token-by-token decode.
- **Precision tests are mandatory.** Communication reordering can preserve shapes while changing numerical accumulation order.
