# vllm-ascend — Ascend 950 QLI Workspace Sizing for Long Context

## What Changed

PR #10041 fixes workspace size and offset calculation for `QuantLightningIndexer` (QLI) on Ascend 950 long-context execution. The PR body says DeepSeek-V4-Flash serving could fail with an MTE DDR address out-of-range error when sequence length exceeded 300K on 950PR.

Only two files changed, but both are on the kernel path:

- `csrc/attention/quant_lightning_indexer/op_host/quant_lightning_indexer_tiling.cpp`
- `csrc/attention/quant_lightning_indexer/op_kernel/arch35/quant_lightning_indexer_kernel.h`

## Why It Matters

Workspace sizing is part of kernel correctness. A QLI kernel can have correct math but still fail if host tiling computes a workspace smaller than the execution path actually indexes. On long-context DeepSeek-V4-Flash, the error appears as an MTE DDR address out-of-range access, which indicates the data movement engine is touching memory beyond the allocated workspace range.

This PR is a compact example of why host tiling logic must be architecture-aware. Ascend 950 uses a different QLI execution path, so reusing the non-950 workspace formula can silently under-allocate.

## Execution Path

1. Host-side QLI tiling computes workspace size and offsets.
2. For Ascend 950, score workspace is sized from aligned `s2Size` using `s1Base=4` and `s2Base=128`.
3. Non-Ascend-950 platforms keep the existing workspace calculation.
4. The `arch35` preload kernel only applies the score workspace offset on AIV, matching the path that actually uses the buffer.
5. MTE copies and kernel memory accesses stay within the allocated workspace.

## Mechanism

The fix separates two concerns that were previously too coupled:

- **Allocation size:** how large the score workspace must be for the Ascend 950 long-sequence path.
- **Offset application:** whether the arch35 preload path actually uses the score workspace offset.

The PR makes both decisions architecture/path-specific rather than using one generic formula.

## Cost Model

This is a correctness fix with performance implications:

| Aspect | Risk Before | Fix |
| --- | --- | --- |
| Workspace size | under-allocation for Ascend 950 long sequence | arch-specific sizing |
| Score workspace offset | applied outside the path that uses it | AIV-gated offset logic |
| Long-context serving | MTE out-of-range failure | workspace matches access pattern |

The PR does not claim a throughput gain. Its value is preventing invalid memory access on a high-value long-context path.

## Trade-offs and Pitfalls

- **Do not generalize the formula blindly.** The new sizing is specifically for Ascend 950 QLI.
- **Architecture gates must match kernel gates.** Host tiling and `arch35` kernel path decisions must stay aligned.
- **Long-context tests matter.** Short sequence tests may never exercise the failing workspace range.
