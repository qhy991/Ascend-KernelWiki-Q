# vLLM Ascend — Async O-Projection TP Weight AllGather in DSA-CP

## What Changed

GitHub PR #10694 proposes asynchronous all-gather support for o-projection tensor-parallel weights in the DSA context-parallel path. The page was observed on 2026-06-19 as an open PR, so this is proposed upstream evidence rather than merged behavior. Targeted diff inspection confirms changes in:

- `vllm_ascend/attention/context_parallel/dsa_cp.py`
- `vllm_ascend/patch/worker/__init__.py`
- `vllm_ascend/patch/worker/patch_process_weights_after_loading.py`

## Why It Matters

Tensor-parallel output projection can become a weight-communication bubble in DSA-CP. If o-projection weights are gathered only at the point of use, the attention pipeline may wait even though earlier stages have independent work available. This PR is a clean example of moving weight communication earlier and making it asynchronous.

## Mechanism

The runtime prepares o-projection TP weights for DSA-CP through worker patching after model weights load. The attention CP path can then initiate or consume the all-gathered weight state around hidden-state/KV preparation, making communication overlap with other DSA prefill work instead of sitting on the critical path.

## Evidence Read

- GitHub PR: `https://github.com/vllm-project/vllm-ascend/pull/10694`.
- GitHub `.diff` file list includes DSA-CP attention code and worker weight-loading patches.
- Agent finding: PR title is `[Feature][Performance] Support o-proj TP weights async allgather in DSA-CP` and it is not already covered locally.

## Trade-offs and Pitfalls

- **Weight readiness becomes stateful.** The model runner must know whether async all-gather has completed before o-proj use.
- **Patch timing matters.** Weight processing after loading must happen before DSA-CP execution starts.
- **Overlap can hide but not remove bandwidth cost.** The benefit depends on enough independent KV/projection work existing after launch.
