# vllm-ascend — Custom AscendC Kernel Support

## Overview

PR #371 ("Custom AscendC op support in vllm_ascend", by ganyi1996ppo) introduces the build scaffolding that lets vllm-ascend ship its own hand-written AscendC kernels rather than relying solely on framework-provided operators. It adds a CMake path that compiles AscendC `.cpp` kernel sources, registers them with PyTorch through `torch_library` bindings so they dispatch on device `"npu"`, and lands `rotary_embedding` as the first custom kernel. A follow-up optimization (issue #802) reports up to **9.36x / 8.26x** speedups for that kernel versus the reference rotary path.

## Why a Custom Kernel Path Matters

vLLM's hot loop in NPU decode is dominated by many small operators applied per token, per step. Routing each of these through generic framework ops incurs dispatch overhead, extra Unified Buffer (UB) round-trips, and missed fusion opportunities. A custom AscendC kernel lets vllm-ascend:

- Fuse the cos/sin gather with the rotate-half application in a single Vector-unit pass, keeping intermediates resident in UB instead of spilling to Global Memory.
- Control MTE (Memory Transfer Engine) data movement explicitly so query/key tiles stream in while the previous tile is still being computed.
- Avoid PyTorch dispatcher overhead in the per-step inner loop, which compounds badly during **multi-step decode** where the same operator fires many times before a synchronization point.

This is the same integration philosophy as the official PyTorch Ascend backend (see `pr-ascend-pytorch-001`), but scoped to vLLM's serving-specific kernels.

## Build Scaffolding (CMake)

The PR adds a `CMakeLists.txt` for the AscendC kernel tree that invokes the Ascend compiler (`bisheng` / `ccec` toolchain via the CANN `ascendc` cmake helpers) to turn each `.cpp` kernel into a device object linked into the extension `.so`. Each kernel source is a standard AscendC operator: a host-side tiling/launch shim plus a `__global__ __aicore__` device entry that runs on the Vector unit.

```cpp
// rotary_embedding.cpp — AscendC device kernel (sketch)
#include "kernel_operator.h"

using namespace AscendC;

class RotaryEmbeddingKernel {
public:
    __aicore__ inline void Init(GM_ADDR query, GM_ADDR key,
                                GM_ADDR cos, GM_ADDR sin,
                                uint32_t rotDim, uint32_t numTokens) {
        // Bind Global Memory tensors and reserve Unified Buffer queues
        qGm.SetGlobalBuffer((__gm__ half*)query);
        kGm.SetGlobalBuffer((__gm__ half*)key);
        cosGm.SetGlobalBuffer((__gm__ half*)cos);
        sinGm.SetGlobalBuffer((__gm__ half*)sin);
        pipe.InitBuffer(inQueue,  BUFFER_NUM, rotDim * sizeof(half));
        pipe.InitBuffer(outQueue, BUFFER_NUM, rotDim * sizeof(half));
    }

    __aicore__ inline void Process(uint32_t numTokens) {
        for (uint32_t t = 0; t < numTokens; ++t) {
            CopyIn(t);     // MTE: GM -> UB
            Compute(t);    // Vector unit: rotate-half + cos/sin fma
            CopyOut(t);    // MTE: UB -> GM
        }
    }

private:
    TPipe pipe;
    TQue<QuePosition::VECIN,  BUFFER_NUM> inQueue;
    TQue<QuePosition::VECOUT, BUFFER_NUM> outQueue;
    GlobalTensor<half> qGm, kGm, cosGm, sinGm;
};

extern "C" __global__ __aicore__ void rotary_embedding(
        GM_ADDR query, GM_ADDR key, GM_ADDR cos, GM_ADDR sin,
        uint32_t rotDim, uint32_t numTokens) {
    RotaryEmbeddingKernel op;
    op.Init(query, key, cos, sin, rotDim, numTokens);
    op.Process(numTokens);
}
```

The `TQue` double-buffering (`BUFFER_NUM` typically 2) lets the next token's `CopyIn` overlap the current token's `Compute`, which is the pipeline-scheduling idea covered in the broader kernel pages.

## torch_library Binding

So that vLLM can simply call the op as a tensor function on an NPU tensor, the PR registers the compiled kernel through PyTorch's library API and provides an `"npu"` (PrivateUse1) implementation. The schema declaration and the device-specific registration look like this:

```cpp
// torch_binding.cpp
#include <torch/library.h>

TORCH_LIBRARY(_C, m) {
  m.def("rotary_embedding(Tensor query, Tensor key, "
        "Tensor cos, Tensor sin, int rot_dim) -> (Tensor, Tensor)");
}

TORCH_LIBRARY_IMPL(_C, PrivateUse1, m) {   // PrivateUse1 == "npu"
  m.impl("rotary_embedding", &rotary_embedding_npu);
}
```

On the Python side this surfaces as a normal custom op, callable in the model's attention preprocessing:

```python
import torch
import vllm_ascend._C  # registers the TORCH_LIBRARY ops

q, k = torch.ops._C.rotary_embedding(query, key, cos, sin, rot_dim)
# query/key live on device "npu"; dispatch lands on rotary_embedding_npu
```

## Performance

The rotary_embedding optimization tracked in issue #802 reports the following speedups over the reference (framework) rotary path on Ascend 910B:

| Path                        | Relative speedup vs reference |
| --------------------------- | ----------------------------- |
| Reference rotary (baseline) | 1.00x                         |
| Custom AscendC kernel       | up to 9.36x                   |
| Custom AscendC kernel       | up to 8.26x (alt. shape/case) |

The two figures (9.36x and 8.26x) correspond to different measured cases reported in #802; both come from keeping the rotate-half computation fused in the Vector unit with UB-resident intermediates rather than materializing them through framework ops. No other numbers are claimed here beyond those reported.

## Trade-offs, Pitfalls, and Notes

- **Maintenance cost.** Each custom kernel is a hand-tuned AscendC source plus a binding; it must track CANN toolchain and PyTorch dispatcher changes. This is justified only for ops that are genuinely hot in the serving loop.
- **Shape coverage.** The reported 9.36x / 8.26x figures are for specific shapes/cases; a custom kernel only wins on the tiling regimes it was tuned for, and may fall back to (or underperform) the reference path outside them. Always validate the actual decode shapes.
- **Device-string correctness.** The implementation must register under `PrivateUse1` (surfaced as `"npu"`); registering under the wrong dispatch key silently routes calls back to the framework op and erases the speedup.
- **UB capacity.** `pipe.InitBuffer` sizes (`rotDim * sizeof(half)` per buffer, times `BUFFER_NUM`) must fit Unified Buffer alongside other resident tensors; oversizing forces spills to Global Memory and negates the fusion benefit.
- **fp16 assumption.** The sketch above assumes `half` I/O, matching the 910B inference target in the frontmatter; other dtypes need separate tiling and are out of scope for this PR.
