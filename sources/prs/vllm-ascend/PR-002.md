# vllm-ascend — Default MoE w2_weight to NZ Format

## Overview

PR #2842 ("[Perf] set moe w2_weight default to be nz" by Angazenn) changes the default storage format of the MoE down-projection weight (`w2_weight`) in vllm-ascend so that it is materialized in `FRACTAL_NZ` once at model load time. The motivation is structural: the Ascend Cube unit consumes operands in `FRACTAL_NZ`, so a weight kept in row-major `ND` must be transposed to `NZ` before every GroupedMatmul call. By pre-converting the expert weights at load, the per-decode-step `ND->NZ` conversion is eliminated from the hot path, removing recurring Vector/MTE work that does no useful arithmetic.

This page documents the change as an upstream-code source and ties it to the format-conversion-overhead anti-pattern (`pattern-format-conversion-overhead`, surfaced concretely in `pattern-nz-format-traps`).

## Why the Cube Unit Wants FRACTAL_NZ

The Ascend Cube unit (`hw-cube-unit`) does not read plain row-major matrices. Its MAC array is fed from L1/L0 buffers in a fractal layout, `FRACTAL_NZ`, where the tensor is tiled into 16x16 (or dtype-dependent) blocks and the blocks are laid out in a "N-major, then Z-inner" order. A logical `[K, N]` matrix in `ND` therefore has to be re-blocked into `NZ` tiles before the Cube engine can stream it. See `technique-nz-tiling` for the tiling geometry and `pattern-nz-format-traps` for the common mistakes.

When a weight is stored as `ND`, the runtime must insert a conversion op (a `TransData`-style reformat) ahead of the matmul. That conversion is pure data movement: it occupies the MTE/Vector pipeline and consumes Unified Buffer bandwidth, but contributes zero FLOPs to the GEMM.

## What Changes in the MoE Path

A Mixture-of-Experts FFN block performs two grouped matmuls per token group:

1. **w1 / gate-up projection**: `hidden -> intermediate` (often already handled in NZ).
2. **w2 / down projection**: `intermediate -> hidden`.

The down-projection weight `w2_weight` is large, is reused on every forward pass, and is read by a `GroupedMatmul` that routes each token group to its expert. Before this PR, `w2_weight` defaulted to `ND`, so each MoE forward re-incurred an `ND->NZ` reformat of the expert weights feeding the Cube unit. This is the repeated-conversion-on-the-hot-path symptom described in `pattern-format-conversion-overhead`.

PR #2842 flips the default so `w2_weight` is stored as `FRACTAL_NZ` from the moment it is loaded.

### Conceptual code: convert once at load, not per step

The grouped MoE matmul on Ascend is driven from Python (vLLM's model-loading and forward layers). The key idea is to move the format conversion out of `forward` and into weight preparation:

```python
import torch
import torch_npu

# --- Load-time weight preparation (runs ONCE per expert weight) ---
def prepare_moe_w2(w2_weight: torch.Tensor) -> torch.Tensor:
    """Materialize the MoE down-projection weight in FRACTAL_NZ once.

    ACL_FORMAT_FRACTAL_NZ == 29 is the Cube-friendly fractal layout.
    Doing this at load time removes the per-forward ND->NZ reformat.
    """
    # torch_npu exposes the CANN TransData reformat via npu_format_cast.
    return torch_npu.npu_format_cast(w2_weight, 29)  # 29 = ACL_FORMAT_FRACTAL_NZ


# --- Hot path (runs EVERY decode step) ---
def moe_down_proj(x_nz, w2_weight_nz, group_list):
    # w2_weight_nz is ALREADY FRACTAL_NZ: no ND->NZ conversion here.
    # GroupedMatmul streams the expert weights straight into the Cube unit.
    return torch_npu.npu_grouped_matmul(
        [x_nz], [w2_weight_nz], group_list=group_list,
        split_item=2,
    )[0]
```

Before the PR, the equivalent of `prepare_moe_w2` effectively happened (implicitly, via a runtime reformat) inside `moe_down_proj` on every call. After the PR, the conversion is amortized to a single load-time operation.

## Cost Model

The savings scale with how often the weight is consumed. For a weight of shape `[E, K, N]` (E experts) the `ND->NZ` conversion has a fixed cost `C` per call; decoding `S` steps turns that into `S * C` of avoidable reformat work in the old path versus `1 * C` in the new path.

| Aspect                        | w2 as ND (before)                      | w2 as NZ (after PR #2842)        |
| ----------------------------- | -------------------------------------- | -------------------------------- |
| Format at load                | `ND`                                   | `FRACTAL_NZ` (converted once)    |
| Conversion frequency          | Every MoE forward (per decode step)    | Once at load                     |
| Hot-path Vector/MTE reformat  | Yes — `ND->NZ` before each GroupedMatmul | No                               |
| Cube unit feed                | Reformatted operand                    | Native `FRACTAL_NZ` operand      |
| Extra load-time work          | None                                   | One `npu_format_cast` per weight |

The exact wall-clock benefit depends on model size, expert count, and decode length; the PR's contribution is qualitative (removing recurring conversions from the per-step path), and no specific speedup figure is claimed here beyond that.

## Trade-offs and Pitfalls

- **Load time and memory layout**: Converting at load adds a one-time `npu_format_cast` per expert weight and may slightly change the in-memory footprint/padding (NZ tiling pads to 16-aligned blocks). This is paid once, not per step.
- **Consistency across the MoE block**: If `w1`/gate-up stays in one format and `w2` in another, make sure any fused or shared code path does not reintroduce a conversion elsewhere. The goal is that *no* `ND->NZ` reformat survives on the decode path — see `pattern-nz-format-traps`.
- **Operands must agree**: `GroupedMatmul`/`npu_grouped_matmul` expects its activation and weight formats to be mutually compatible for the Cube unit. Pre-converting only the weight is correct when the activation is already produced in (or cheaply consumed as) a Cube-compatible layout; verify the activation side does not silently force a fallback reformat.
- **Don't double-convert**: Once `w2_weight` is `FRACTAL_NZ`, ensure no downstream code calls another reformat "just in case" — that would re-add the overhead this PR removes. This is the canonical trap in `technique-format-conversion`.
- **Architecture scope**: This change targets Ascend 910B Cube-unit semantics. The fractal-block geometry is hardware-specific; treat the `29 == ACL_FORMAT_FRACTAL_NZ` mapping as the CANN constant, not a portable assumption.
