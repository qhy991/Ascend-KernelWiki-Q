# vllm-ascend — DeepSeek-V4 Support for Ascend 950

## What Changed

PR #9757 is the main DeepSeek-V4 enablement PR for Ascend A5 / Ascend 950 in vllm-ascend. The PR body lists three major areas: W4A8 MXFP quantization, Ascend A5 attention kernel optimizations, and DeepSeek-V4 model/runtime integration.

The change spans 27 files (+1407/-557). It touches attention, device operators, DeepSeek-V4 model classes, fused MoE, linear ops, RoPE, platform detection, quantization methods, compressor patches, and the v1 model runner.

## Why It Matters

This PR is a model-porting umbrella for a new hardware/model combination. It is not just a config flag. DeepSeek-V4 on Ascend 950 requires changes across:

- **Quantization:** W4A8 MXFP and FP8 configuration/method adapters.
- **Attention:** DSA v1 path and A5-specific attention scheduling/KV interfaces.
- **MoE:** fused MoE and grouped matmul / swish-glu operator integration.
- **Runtime:** model runner and platform feature handling.
- **Compression:** DeepSeek compressor and KV/cache-related patches.

That makes it a high-value source page for later topic pages about DSV4 on Ascend.

## Execution Path

A simplified runtime path is:

1. DeepSeek-V4 model loading selects Ascend quantization configuration via `fp8_config.py` and method adapters.
2. Linear and MoE layers route W4A8/MXFP computation through device operator methods in `device_op.py`.
3. DSA attention is handled through `attention/dsa_v1.py` and `ops/dsa.py` with A5-specific metadata and KV-cache interfaces.
4. Fused MoE code adapts runtime arguments, stage contracts, communication method, and MLP execution.
5. `model_runner_v1.py`, platform patches, and model classes coordinate the end-to-end serving path.

## Mechanism

The PR combines several mechanisms:

- **W4A8 MXFP quantization:** introduces or updates quantization methods for float4 weight formats and float8 microscaling metadata.
- **A5 DSA attention:** adapts block sizes, scheduling metadata, and KV interfaces for Ascend 950 execution.
- **Device op dispatch:** centralizes hardware-specific quantized matmul and grouped matmul-swish-glu calls.
- **Model runner integration:** ensures the serving loop can actually invoke the new model/quant/attention paths.

## Cost Model

The expected gains come from matching DeepSeek-V4's model structure to Ascend 950 execution primitives:

| Area | What the PR Enables | Why It Matters |
| --- | --- | --- |
| Quantization | W4A8/MXFP methods and config | lower bandwidth and storage for large model weights |
| Attention | DSA v1 A5-specific path | long-context sparse attention on target hardware |
| MoE | fused MoE and grouped matmul swish-glu hooks | fewer framework fallbacks in expert layers |
| Runner | model/platform patches | makes the optimized paths reachable in serving |

## Trade-offs and Pitfalls

- **This is an umbrella PR.** Individual mechanisms need separate source pages or technique pages for exact kernel details.
- **Ascend 950 is not in the current architecture vocabulary.** Frontmatter uses `ascend910b` for schema compatibility while preserving Ascend 950 in title/tags/body.
- **Quant scale semantics are high risk.** W4A8 MXFP paths must align weight format, scale format, and device op expectations.
- **DSA follow-up fixes matter.** Later PRs such as QLI workspace sizing and piecewise graph handling refine this base support.
