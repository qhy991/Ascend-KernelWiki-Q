# vllm-ascend — Trim SWA KV Transfer Blocks Before Window Clipping

## What Changed

PR #10255 fixes the order of block selection for sliding-window-attention (SWA) KV transfer in PD-disaggregated serving. The PR body says the previous code clipped SWA blocks first and only then trimmed transfer blocks by actual prompt length. That ordering could leave stale or unwritten SWA tail blocks in `remote_block_ids`.

The patch changes `vllm_ascend/distributed/kv_transfer/kv_p2p/mooncake_hybrid_connector.py` and adds a focused unit test in `tests/ut/kv_connector/test_mooncake_hybrid_connector.py`.

## Why It Matters

This is a correctness fix for KV-cache transfer semantics. In a PD-disaggregated setup, the prefill side may allocate block IDs for the original prompt length. If stale SWA tail blocks are transferred, the decode side can consume unwritten data in `npu_sparse_attn_sharedkv`. The PR body reports that layer-0 attention can then produce all-NaN hidden states and the request can later get stuck.

The important lesson is ordering: first trim transfer block IDs by the actual prefill-side prompt length, then apply the SWA window clipping.

## Execution Path

1. Prefill side computes transfer block IDs for KV cache groups.
2. SWA groups may have block IDs corresponding to the original prompt length.
3. The fix trims transfer block IDs by actual prompt length for all KV cache groups.
4. Non-SWA compressed groups preserve their compressed token length handling.
5. SWA window clipping is applied after prompt-length trimming.
6. Decode side receives only valid remote SWA blocks for `npu_sparse_attn_sharedkv`.

## Mechanism

The test case described in the PR captures the bug precisely:

- `prompt_len=129`
- SWA block IDs: `[100, 101, 102, 103]`
- `num_swa_blocks=2`
- expected remote SWA blocks: `[100, 101]`
- stale tail blocks `[102, 103]` must not be sent

This is a block-list transformation bug, not a sparse attention math bug. The sparse attention kernel only exposes the issue because it consumes the wrong KV blocks.

- Stated regression case: prompt length 129, SWA block list `[100, 101, 102, 103]`, expected `[100, 101]`.

## Cost Model

The fix does not optimize a kernel loop. It prevents wasted and unsafe KV transfer:

| Step | Bad Ordering | Correct Ordering |
| --- | --- | --- |
| 1 | clip SWA window | trim by actual prompt length |
| 2 | trim transfer blocks | preserve compressed-group handling |
| 3 | stale tail can remain | clip SWA window after trim |
| Result | NaN/stuck risk | only valid remote blocks sent |

## Trade-offs and Pitfalls

- **KV block IDs are not self-validating.** A block ID can point to allocated but unwritten or stale cache.
- **SWA and compressed groups differ.** Fixes must preserve compressed token length behavior for non-SWA groups.
- **NaN root cause is upstream of attention.** Debugging only the sparse attention kernel would miss the transfer-ordering bug.
