# vllm-ascend — AscendC `store_kv_block` Op (Scatter into Paged KV Cache)

## Overview

PR #10292 ("[Feature] add ascendc ops store_kv_block" by ZT-AIA) adds a custom AscendC operator, `store_kv_block`, to vllm-ascend. Its job is narrow but latency-critical: take a batch of freshly computed key vectors and write them into the right slots of a **paged KV cache** held in Global Memory, where each token group may land at an arbitrary, indirection-resolved cache block offset. The operator is gated behind `additional-config` flags `enable_sparse_c8` + `c8_enable_reshape_optim` and is used only on the P (prefill) node.

This page documents the operator as an upstream-code source. It is a compact, readable example of three recurring AscendC themes: (1) **multi-core work distribution** with a tail core, (2) **GM→UB→GM scatter** through the Unified Buffer, and (3) **explicit MTE2/MTE3 event synchronization** between the load and store memory engines. It ties directly to `technique-async-dma-multistage`, `technique-multicore-grid-scheduling`, and `pattern-format-conversion-overhead` (the broader KV-cache paging story in `technique-kv-cache-paging`).

## Why a Custom Kernel for KV Writes

In a paged attention system the KV cache is not a contiguous `[num_tokens, hidden]` tensor. Each sequence's K/V tensors are scattered across a pool of fixed-size blocks addressed by a block table; a token group with logical index `g` must be written to `keyCache[ groupKeyCacheIdx[g] * tokenSize ]`, where the index comes from indirection tables computed earlier in the step.

Doing this scatter with generic PyTorch ops (`index_copy`, `scatter`, advanced indexing) launches several kernels, materializes intermediates, and pays Python/dispatch overhead per step — none of which do useful arithmetic. The AscendC kernel folds the whole scatter into a single multi-core launch that streams each group through on-chip UB and writes it straight back to GM. The compute is trivial (it is pure data movement); the value is in removing host-side dispatch and minimizing GM round-trips.

## Operator Inputs and Tiling

The kernel entry point (`store_kv_block`) takes six GM buffers:

| Argument | Role |
| --- | --- |
| `keyIn` | Source keys (this step's freshly computed K), `[numTokens, ...]` |
| `keyCacheIn` | Destination paged KV cache in GM |
| `groupLen` | Per-group byte length to copy |
| `groupKeyIdx` | Source row index for each group (into `keyIn`) |
| `groupKeyCacheIdx` | Destination block index for each group (into `keyCacheIn`) |
| `tiling` | Host-computed tiling blob |

The host-side tiling (`store_kv_block_tiling.cpp`) does the platform-aware work:

- Reads the platform core count (`ascendcPlatform.GetCoreNum()`) and distributes the `groupInfoLen` groups across cores as `corePerNum = groupInfoLen / coreNum`, with `coreTail = groupInfoLen % coreNum` cores getting one extra group.
- Derives `tokenSize = numHeads * prod(headSize)` and `blockTableSize = blockSize` (the page block size, from an op attribute).
- Picks `tilingKey` from the dtype: `1` → int8/uint8, `2` → fp16/bf16, `4` → int32/uint32. The kernel uses this to instantiate the right `StoreKVBlockBase<T>`.
- Guards the per-page element count against `MAX_UB_USE_SIZE = 180 * 1024` bytes so a single page block always fits in UB.

This is the standard AscendC host-tiling split: **shape/platform logic on the host (CPU), per-core loop in the kernel (AICore)**.

## Multi-Core Work Distribution

The kernel distributes groups across cores with a **stride-`blockNum` (cyclic) scheme** rather than a contiguous split:

```cpp
coreId = AscendC::GetBlockIdx();
blockNum = AscendC::GetBlockNum();
if (coreId < coreTail) {
    corePerNum = tilingData->corePerNum + 1;   // first coreTail cores get one extra
} else {
    corePerNum = tilingData->corePerNum;
}
...
for (uint32_t i = 0; i < corePerNum; i++) {
    uint32_t idx = coreId + i * blockNum;       // cyclic, not contiguous
    ...process group idx...
}
```

Two things are worth noting:

- **Cyclic vs. contiguous.** Core `c` handles groups `c, c+blockNum, c+2*blockNum, …`. Adjacent groups go to different cores. For a scatter where group lengths vary, this spreads long and short groups across cores more evenly than a contiguous split, reducing tail imbalance.
- **Tail handling.** Only the first `coreTail` cores get the extra group; the host explicitly sets `coreTail` (and the code comments that when `corePerNum == 0`, idle cores beyond `coreTail` touch no address). This avoids the classic off-by-one where a naive `ceildiv` makes every core fetch one phantom group.

This is the same shape as `technique-multicore-grid-scheduling`: derive `(perCore, tail)` on the host, branch in the kernel.

## The Scatter Loop: GM → UB → GM with Event Sync

Each group is copied in two `DataCopyPad` hops through a single UB tensor (`tokenLocal`), sized `blockTableSize * tokenByteSize`:

```cpp
pipeThis->InitBuffer(tokenBuf, blockTableSize * tokenByteSize);
tokenLocal = tokenBuf.Get<T>();

AscendC::DataCopyExtParams copyParams{1, 0, 0, 0, 0};
AscendC::DataCopyPadExtParams<T> padParams{false, 0, 0, 0};

for (uint32_t i = 0; i < corePerNum; i++) {
    uint32_t idx = coreId + i * blockNum;

    copyParams.blockLen = groupLenGt.GetValue(idx) * tokenByteSize; // bytes

    // GM(keyIn) -> UB
    DataCopyPad(tokenLocal,
                keyInputGt[groupKeyIdxGt.GetValue(idx) * tokenSize],
                copyParams, padParams);
    AscendC::SetFlag<AscendC::HardEvent::MTE2_MTE3>(EVENT_ID1);
    AscendC::WaitFlag<AscendC::HardEvent::MTE2_MTE3>(EVENT_ID1);

    // UB -> GM(keyCache)
    DataCopyPad(keyCacheInputGt[groupKeyCacheIdxGt.GetValue(idx) * tokenSize],
                tokenLocal, copyParams);
    AscendC::SetFlag<AscendC::HardEvent::MTE3_MTE2>(EVENT_ID1);
    AscendC::WaitFlag<AscendC::HardEvent::MTE3_MTE2>(EVENT_ID1);
}
```

The key AscendC concepts visible here:

### Two memory engines, two event flags

Ascend cores have separate memory-transfer engines: **MTE2** loads GM→UB (the "in" DMA), **MTE3** stores UB→GM (the "out" DMA). They run on independent instruction queues (see `hw-instruction-queue`), so they *can* overlap — but only if you tell the hardware about the dependency. That is what `SetFlag`/`WaitFlag` do:

- After the GM→UB `DataCopyPad`, `SetFlag<MTE2_MTE3>` + `WaitFlag<MTE2_MTE3>` enforces "the load of `tokenLocal` is visible before the store reads `tokenLocal`." Without it, the MTE3 store could fire before MTE2 finishes filling UB and read stale data.
- After the UB→GM store, `SetFlag<MTE3_MTE2>` + `WaitFlag<MTE3_MTE2>` returns the token to the "loadable" state, so the next iteration's GM→UB load does not race with the previous store's read of `tokenLocal`.

This is the explicit, flag-based equivalent of CUDA's `cp.async` barrier / `barrier.sync`. See `technique-async-dma-multistage` and `hw-event-sync`. Because both flags use `EVENT_ID1`, the kernel uses one event slot for the whole ping-pong; it does **not** double-buffer here (the `WaitFlag<MTE3_MTE2>` at the end of each iteration serializes the next load behind the previous store).

### `DataCopyPad` for unaligned, variable-length copies

`copyParams.blockLen = groupLen * tokenByteSize` is set **per group** inside the loop, so each group can be a different length. `DataCopyPad` (not plain `DataCopy`) is used because group lengths are generally not 32-byte aligned and may not tile cleanly; the pad variant handles the tail bytes safely (`padParams{false, 0, 0, 0}` = no padding, just safe edge handling). This is the right primitive for indirection-driven scatter where alignment cannot be assumed.

### Single UB buffer, no double-buffer

`InitBuffer` allocates exactly one `blockTableSize * tokenByteSize` UB region. The loop is load→wait→store→wait, serialized. This is correct and simple but leaves DMA bandwidth on the table: while MTE3 stores group `i`, MTE2 could be loading group `i+1`. The natural next optimization (not in this PR) is a two-element UB ping-pong and moving the `MTE3_MTE2` wait to the *top* of the next iteration, turning the serialized loop into the `technique-double-buffering` / `technique-async-dma-multistage` pipeline. The current single-buffer form is a clean baseline.

## Cost Model

The operator moves `Σ_g groupLen[g] * tokenByteSize` bytes per step: one GM read + one GM write per byte, plus UB residency. No FLOPs. Its benefit versus a Python scatter is purely in:

1. **One kernel launch** instead of N indexing/scatter ops and their intermediates.
2. **Multi-core parallelism** across all AICores, each streaming through its own UB.
3. **No host round-trip** per group — indices are read from GM tables inside the kernel.

The savings scale with the number of token groups per prefill step; the PR scopes it to the P node under the sparse-C8 reshape-optim path, where the group count is high enough to make the custom kernel worthwhile.

## Trade-offs and Pitfalls

- **UB capacity gate.** The host rejects `pageBlockEleSize > MAX_UB_USE_SIZE (180 KiB)`. If you raise the page block size, the single-buffer allocation will overflow UB and the op fails at tiling — you must either shrink the block or move to a tiled (multi-pass) copy.
- **No bounds checking in the kernel.** The commented-out guard (`groupLen <= 0`, negative indices) is disabled. Indirection tables must be valid; a bad `groupKeyCacheIdx` writes to an arbitrary GM offset. The host must guarantee table correctness (the PR relies on the caller's metadata).
- **Single event slot, serialized DMA.** As noted, this is not pipelined. If this op shows up as a P-node bottleneck, the fix is a double-buffered UB with two event IDs, not more cores.
- **Type coverage via `tilingKey`.** Only int8/fp16/bf16/int32 are supported. FP8 (`float8_e4m3fn`) — increasingly relevant for quantized KV — is not wired up; adding it means a new `tilingKey` branch and a `StoreKVBlockBase` instantiation, plus verifying `DataCopyPad` semantics for the FP8 element size.
- **P-node only.** The op is gated to prefill under sparse-C8. On the decode path the per-step group count is small and a custom kernel is not worth the launch overhead — a generic scatter is fine there.
