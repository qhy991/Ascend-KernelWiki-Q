# vllm-ascend — Layerwise KV Pooling for AscendStore

## What Changed

PR #10077 adds Layerwise KV Pooling to vllm-ascend's AscendStore KV-transfer stack. The PR body says the old layerwise design suffered from per-layer key growth, metadata lookup latency, and Python-side HBM address computation overhead.

The patch is large: 29 files, +5439/-959. It includes a user guide, pool scheduler/worker changes, KV-transfer logic, config data updates, and extensive unit tests.

## Why It Matters

Bulk KV transfer serializes model execution: the full KV cache must be saved after a forward pass or loaded before work can continue. Layerwise transfer changes this into a pipeline. A producer can save layer `i` after its attention finishes while later layers compute; a consumer can wait only for the current layer while later layer transfer overlaps with compute.

This is a system-level version of kernel pipelining: the tile is a transformer layer rather than a UB block.

## Mechanism

Layerwise mode separates save/load by layer:

- producer / `kv_both`: send each layer's KV after that layer computes;
- consumer / `kv_both`: wait for layer `i` before computing layer `i`, while later layers can transfer;
- metadata is moved away from a naive per-layer/per-block key explosion toward pool/request-level information plus derived ranges.

## Trade-offs and Pitfalls

- Layerwise transfer requires strict per-layer ordering: compute must not consume KV before that layer has arrived.
- Partial transfer failures need finer-grained invalidation than bulk transfer.
- Benefits are largest when KV transfer latency can be overlapped with enough layer compute.
- Metadata and address calculation must remain group-aware for hybrid/Mamba cache layouts.
