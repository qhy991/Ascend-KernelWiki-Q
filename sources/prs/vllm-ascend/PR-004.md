# vllm-ascend — W8A8FP8 Dynamic Quantization on Ascend 950

## Overview

PR #10236 ("[Ascend950][Feature] Add support for W8A8FP8 dynamic quantization on Ascend 950" by herizhen) introduces the `W8A8FP8` quantization type to vllm-ascend and implements the matching linear and fused-MoE schemes: `AscendW8A8FP8DynamicLinearMethod` and `AscendW8A8FP8DynamicFusedMoEMethod`. Activations are dynamically quantized to **FP8 (`float8_e4m3fn`) per token**, weights are stored as **FP8 per channel**, and the matmul runs through CANN's fused `npu_quant_matmul` on the Cube unit.

This page documents the PR as an upstream-code source. The interesting part is not new kernel code — it is how an existing CANN quantized-matmul primitive (`npu_dynamic_quant` + `npu_quant_matmul`, shared with the int8 W8A8 path) is *re-pointed* at FP8 operands by swapping dtypes and scale layouts, and where the FP8 path has to diverge from int8.

## W8A8 vs W8A8FP8: What Changes

Both schemes are "dynamic": the activation is quantized on the fly per forward pass (per token), and the weight is quantized once at load time (per channel). The PR implements W8A8FP8 as a **subclass** of the existing int8 W8A8 method, overriding only what differs:

| Aspect | W8A8 (int8) | W8A8FP8 (#10236) |
| --- | --- | --- |
| Activation dtype | `torch.int8` | `torch.float8_e4m3fn` |
| Weight dtype | int8 | `torch.float8_e4m3fn` |
| Activation quant | per-token, dynamic | per-token, dynamic |
| Weight quant | per-channel, load-time | per-channel, load-time |
| Act scale | int64-encoded fp32 (`scale_from_float_to_int64`) | fp32 `pertoken_scale` |
| Bias in matmul | fused via `npu_quant_matmul(..., bias=bias)` | **not** fused — added after, see below |

The activation quantization step is identical in shape for both:

```python
quantized_x, pertoken_scale = torch_npu.npu_dynamic_quant(x, dst_type=self.act_quant_type)
```

`act_quant_type` is the only real switch: `torch.int8` for W8A8, `torch.float8_e4m3fn` for W8A8FP8. `npu_dynamic_quant` is the CANN op that, in one launch, computes the per-token scale and produces the quantized activation tensor — fusing the max-abs reduction, scale division, and cast that a naive implementation would do as three separate kernels.

## The Fused Quantized Matmul

Both schemes feed the quantized operands into the same primitive:

```python
output = torch_npu.npu_quant_matmul(
    quantized_x,          # per-token quantized activation
    layer.weight,         # per-channel quantized weight (FP8 here)
    layer.weight_scale,   # per-channel weight scale
    pertoken_scale=pertoken_scale,   # per-token activation scale
    bias=bias if self.act_quant_type == torch.int8 else None,
    output_dtype=x.dtype,
)
```

`npu_quant_matmul` is the CANN fused kernel that runs on the **Cube unit**: it dequantizes operands on the fly (applying the per-token and per-channel scales inside the MAC accumulation), accumulates in higher precision, and writes back in `output_dtype` (typically bf16/fp16). Doing the dequant *inside* the matmul — rather than dequantizing to fp16 first and then calling a plain matmul — is what keeps the Cube unit fed with low-precision operands and avoids a full-precision intermediate tensor. This is the Ascend analogue of CUTLASS/`_scaled_mm`-style fused FP8 GEMM, and it is the core of `technique-operator-fusion` applied to quantized linear.

The PR also supports the **chunked gate-up** path used by gate-up-merged linears (and by the MoE w1/w3 projection): when `_chunk_size > 0`, it splits the weight into `weight_1`/`weight_2` halves and issues two `npu_quant_matmul` calls concatenated along the output dim — same primitive, twice, for the SwiGLU gate and up projections.

## Where FP8 Has to Diverge from int8

The one place W8A8FP8 cannot just inherit int8 behavior is bias fusion:

```python
def apply(self, layer, x, bias=None, tp_rank=0):
    output = super().apply(layer, x, bias, tp_rank)   # bias=None on FP8 path
    # TODO: there is a bug in npu_quant_matmul for fp8 with bias
    # after the bug is fixed, the whole apply method can be removed.
    if bias is not None:
        output = (output + bias).to(x.dtype)
    return output
```

On the int8 path, `npu_quant_matmul(..., bias=bias)` fuses the bias add into the Cube epilogue. On the FP8 path that fused-bias branch has a CANN bug, so the PR computes the matmul **without** bias and adds it as a separate elementwise op afterward (`output + bias`). This is the pragmatic compromise: correctness first, with an explicit TODO to re-fuse once CANN fixes the FP8+bias epilogue. The cost is one extra Vector pass over the output and one extra GM round-trip for the bias add — small relative to the matmul, but a real regression versus the int8 path until the bug is fixed.

## Weight Layout: Per-Channel Scales and Offsets

Weights are prepared once at load time. For FP8 the per-channel parameters are:

```python
def get_perchannel_param(self, output_size, params_dtype):
    return {
        "weight_scale":  torch.empty(output_size, 1, dtype=torch.float32),
        "weight_offset": torch.empty(output_size, 1, dtype=params_dtype),
    }
```

`weight_scale` is fp32 and shaped `[output_size, 1]` — one scale per output channel (column of the weight). `weight_offset` carries the per-channel zero point. The `[N, 1]` shape broadcasts cleanly against the `[N]` output dimension inside `npu_quant_matmul`. For the MoE case the same pattern is repeated per expert and per projection (`w13_weight_scale`/`w2_weight_scale`), so each expert's gate-up and down-projection weights carry their own per-channel scales.

This per-channel (axis=0, output) layout is the one `npu_quant_matmul` expects; matching it is what lets the FP8 weight feed the Cube unit without a runtime transpose or reformat. Note the connection to `pr-vllm-ascend-002` (MoE `w2` defaulting to NZ): both PRs are about getting weights into the layout the Cube unit consumes *once*, at load, rather than per step.

## MoE Integration

`AscendW8A8FP8DynamicFusedMoEMethod` extends the int8 fused-MoE method, overriding `get_weight` and `get_dynamic_quant_param` to allocate FP8 expert weights (`w13_weight`, `w2_weight` as `float8_e4m3fn`) with matching per-expert, per-channel scales. The dispatch/token-routing plumbing (`token_dispatcher.py`, `moe_runtime_args.py`, `moe_stage_params.py`) and the `QuantType.W8A8FP8` enum entry are the glue that lets the MoE MLP (`moe_mlp.py`) pick this scheme when the model config requests FP8-dynamic. The activation-side `npu_dynamic_quant` is applied per token before the grouped matmul, exactly as in the linear case.

## Cost Model and Trade-offs

- **Memory.** FP8 weights are half the size of int8 (already 8-bit) and a quarter of bf16 — the win versus the int8 W8A8 path is essentially zero in weight memory (both 8-bit), but FP8 often preserves more accuracy than int8 at the same bit width, so it trades accuracy headroom for the same bandwidth.
- **Compute.** The Cube unit consumes 8-bit operands either way; the throughput difference between int8 and FP8 quantized matmul is implementation-dependent in CANN. The PR does not claim a speedup number — its contribution is *enabling* FP8-dynamic on Ascend 950, not benchmarking it.
- **Bias fusion regression.** Until the CANN FP8+bias bug is fixed, FP8 linears with bias pay an extra elementwise add. For bias-free projections (most attention/MoE internal matmuls) there is no regression.
- **Per-token dynamic cost.** `npu_dynamic_quant` runs every forward; for decode (batch 1) this is cheap, for large prefill batches the per-token reduction is the cost. This is inherent to *dynamic* quantization — static schemes avoid it but lose per-token accuracy.

## Pitfalls

- **Don't dequantize-then-matmul.** The whole point is that `npu_quant_matmul` applies scales inside the Cube unit. A "safe" refactor that dequantizes `quantized_x` to bf16 first and calls a plain matmul destroys the bandwidth benefit and is the anti-pattern in `pattern-format-conversion-overhead`.
- **Scale dtype/encoding.** The int8 path re-encodes the fp32 scale into int64 (`scale_from_float_to_int64`) for `npu_quant_matmul`; the FP8 path passes fp32 `pertoken_scale` directly. Mixing the two encodings silently breaks the matmul. Keep the scale representation matched to `act_quant_type`.
- **Architecture scope.** The PR is labeled Ascend 950 (the post-910B generation). FP8 support on the Cube unit is generation-dependent; do not assume `float8_e4m3fn` `npu_quant_matmul` works unchanged on 910B — verify the CANN version and platform.
- **FP8+bias TODO.** Anyone inheriting this scheme must carry the workaround until CANN ships the fix, or bias will be wrong on the FP8 path.
